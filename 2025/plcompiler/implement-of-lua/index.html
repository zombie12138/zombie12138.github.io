小而美的 Lua 实现 | Implement of Lua

发表于2025-02-04|更新于2025-12-23|PL&Compiler
|总字数:1.8k|阅读时长:6分钟|浏览量:

最近看了一个 Lua 相关的项目, 对于 Lua 很感兴趣. 看一下 Lua 是怎么实现的. 主要参考 Lua 的论文: The Implementation of Lua 5.0[1]. ## Lua - Lua 的设计理念, 也是 Lua 能广泛应用的原因: simple, efficient, portable, and lightweight. - Lua 的重大变更: 5.0 coroutines, 5.1 incremental garbage collection 一些 Lua 5.0 相比于 Lua 4.0 新增的东西 - Register based virtual machine - New algorithm for optimizing tables used as arrays - The implementation of closures - The addition of coroutines Lua 虚拟机最初是手写的 Scanner 和 Parser, 后来 3.0 换成了 yacc. 并且, Lua 没有独立的 IR, 是 parse 之后直接 emits instructions for the virtual machine on the fly 此外 Lua 采用 while-switch dispatch loop, 而非 direct threaded code, 以保留更好的兼容性. 两者的示例大概如下, DTC 使用到了非标准的 C 拓展[2][3][4]:

## Lua 的数据表示 Lua 包含几种基本类型: `nil`, `boolean`, `number`, `string`, `table`, `function`, `userdata` (include heavy and light), and `thread` Lua 的数据存储为 tagged unions, 这种设计兼顾可移植性, 空间和性能: 和很多语言的设计一样, lua 中的 `internalizes string` 是存在 `hash` 表中, 并且 `string` 都是 `immutable` 的. 对于很长的 String, 为了节约性能, Lua Hash 不会去关注每一个 Byte. ## Lua table `Table` 是 `Lua` 中的唯一的数据结构, 其性能对于 Lua 至关重要. Lua 中的 `Table` 是一个 `associative arrays`, 可以使用 `nil` 外任何数据来作为 key. 其实 Lua 中的 `table` 就是 array + hashmap, 实现也是. 这种实现有以下好处: 1. 是不需要维护两种数据结构, 保持了本身的紧凑; 2. 用户不需要做选择; 3. 对于稀疏的数组更加友好; 4. 对于存在数组中的元素访问速度会很快 4.0 之前, Lua 的 `Table` 全部都在哈希表中, 5.0 之后, lua 带来了一种上图中的混合的数据结构, 包含一个 Array 和一个 HashMap. Lua 会动态调整这两个部分, 两者可以同时存在. 数组和哈希部分都可以扩展, 其中数组会保证新的大小 n 至少有一半 slot 在使用, 并且 `[n/2, n)` 存在元素. 哈希部分则是使用的 Brent's Method 的变体, 具体可以参考这个论文[5] 具体的实现可以看一下 Lua Table 分析的源代码. ## Lua 的函数和代码 Lua 在编译函数的时候, 都会创建一个包含机器指令, 变量和调试信息的 `prototype`. Lua 执行 `function...end` 语句的时候, 就会产生一个闭包 (`closure`), 包含 `prototype` 的引用, 一个用于查询全局变量的 `environment`, 以及一个 `upvalue` 的数组. 除了 C 大部分语言都有闭包的概念, 解决的就是下面, 函数引用外层的局部变量的问题: 大部分语言处理闭包都加了一些限制, 比如 python 无法引用一个可变的外层局部变量, 除非使用了 `nonlocal`; 而一些语言则采用了更复杂的机制, 但这与 Lua 追求简洁的设计理念相悖 Lua 使用了一种 `upvalue` 的东西来实现的闭包. 首先闭包的变量可能存活在一个栈上 (上图左边所示), 这个 `upvalue` 会引用到栈上的变量; 当栈上变量脱离作用域的时候, 则会拷贝到 `upvalue` 中. `Mutable` 的状态在闭包中仅仅创建一次, 所有的 `upvalue` 串成一个链表 (图上 pending vars 的部分), 插入新的 `upvalue` 之前, 先遍历链表查看是否存在可复用的 `upvalue`. 如果引用外层的外层函数, Lua 使用 Flat Closure 来解决这个问题. 当一个函数引用了外层的外层的变量, 这个变量也会加入外层的闭包. 因此一个函数实例化的时候, 闭包中的变量要么来自于外层的 stack, 要么来自直接外层的函数的 closure. ## Coroutines 5.1 之后, Lua 实现了一个 asymmetric coroutines / semi-coroutine, 实现包含三个 coroutine 下的标准库函数: `create`, `resume`, and `yield`. `create` 创建返回一个 `thread`; `resume` 执行; `yield` 返还控制权. 一个关键是不能使用 C 语言的 stack 来实现, 也叫 stackless interpreter. 这里有点没看懂, LLM 的解释是, Lua 使用 C 栈来保持调用关系, 但是 `yield` 返回的时候除了退出之外, 还保留了虚拟栈的上下文, 下次调用的时候直接恢复. ## Lua VM Lua 会编译源码到 `opcodes`, 每个函数 Lua 会编译成成一个 prototype, 包含 Lua 的 `opcodes` 的数组和一个常量的数组 (Array of TObjects). 5.0 之前 Lua 使用的是 stack based vm, 之后使用的是 register based vm. 但是 register based VM 也会使用 stack 来分配 activation records, 包含局部变量, 返回值等. 不同是避免了 push, pop 这些 Lua 中昂贵的操作. register based VM 对于内存访问效率更高, 对于 JIT 也更加友好; 但是带来的问题就是 指令操作数更大 (指令数也会少一些), 解码也更加费力 (但是无需计算隐式参数, 如栈顶; 同时也某种程度避免了内存对齐等问题的开销). Lua VM 共有 35 个指令. 一条指令 32 bit, 6b 来存放指令类型, 剩下 A, B, C 8+9+9 是操作数. A 一般为返回值结果的寄存器, B 和 C 为操作数 (register / constant), 这样的设计可以使用更少的指令. 一条跳转指令需要容纳两个操作数和一个跳转偏移量, 如果将这些都放在单条指令中, 空间会不足, 只能 9 bit 的跳转长度. 因此 Lua 的跳转也是一个 test 跟着一个跳转指令. 下面可以看到 Lua 4.0 和 Lua 5.0 实现相同功能, 指令的差距: ## 总结 Lua 是第一个大规模使用 register based VM 的语言, Lua 的闭包, Lua 的 `table` 也确实非常 unique. 作者用 [The Great Computer Language Shootout](https://en.wikipedia.org/wiki/The_Computer_Language_Benchmarks_Game) 做了一个性能比较, 得益于新的 VM, Lua 的运行时间最多减少了 56%. ## 参考文献
