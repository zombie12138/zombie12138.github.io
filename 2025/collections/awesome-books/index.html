技术书评手记

发表于2025-03-08|更新于2026-04-28|Collections
|总字数:2.9k|阅读时长:8分钟|浏览量:

决心要多读书，但是最终应该也看不了几本，吐槽和感受就放这里。

Contents

从下面的几个维度评判

  • 书名
  • 标签：书本内容，例如：Java、Performance
  • 深度：对于一个技术细节，是否完全深究、弄清楚
  • 广度：是否会谈论起更大的话题，或者是更多的横向比较
  • 易读：是否讲的是人话
  • 正确：翻译，以及原文中，是否存在事实性错误
  • 难度：整体的难度，包括厚度、所需知识、深度、易读性等。
  • 推荐：综合推荐评分

Java 性能优化实践

这本动物书主要讲了 Java8 的性能优化实践。当时是在美团博客的参考链接看到的，什么也没想直接买了。

内容评价

打开之后其实还是比较失望的，主要是讲得并不够深入。比如 Java 中影响性能最大的因素之一——垃圾回收，虽然讲解了原理，但是没有深入谈论细节，比如 G1 的每一个步骤具体是做什么等；对于性能调优的常见方法讲得也非常浅；至于工具，我认为这个应该只是用的时候去查询，对于工具的实现原理也没有认真讲。虽然这是一本实践类书籍，但是我认为不应该讲太多工具，应该深入原理，并且引申出来一些模式，而不是简单地罗列工具。

事实问题

最让我生气的是文中的一个反模式，提到 艾尔斯伯格悖论。最终作者得到的结论是，人们对于概率估计总是有误差。

这个实验大概是这样：箱子中有 90 个球，30 个红球，其余 60 个要么是黑球，要么是黄球。四个赌局：

  • A：如果抽到红球，获得 100 元，否则 0 元
  • B：如果抽到黑（Black）球，获得 100 元，否则 0 元
  • C：如果抽到红球或黄球，获得 100 元，否则 0 元
  • D：如果抽到黑球或黄球，获得 100 元，否则 0 元

实验表明，人们在 AB 之间更趋向于选择 A（固定的 1/3 概率获胜），而不是 B（以 0~2/3 概率获胜）；在 CD 之间更倾向于选择 D（固定的 2/3 概率获胜），而不是 C（以 1/3~1 的概率获胜）。选择 A 的一个潜在判断是黄球（比黑球）更多，选择 C 的潜在判断是黄球更少。因此排除了人们本身判断的变量，得出的结论是人们 更喜欢消除不确定性，这个指的是不确定的概率。

但是书中的结论是人们总是 对于概率有错误的估计。但是整个实验我 无法做出黑球还是黄球数量多少的判断。虽然有逻辑上的矛盾，但是这种矛盾是人类对不确定性的厌恶导致的，并没有文中说的对于概率有错误的预估。

如果说人们对于概率的直觉更差的实验，应该是主持人开门问题。三个门只有其中一个后面藏有宝藏，你可以选择一个，但是主持人会打开另外两个门里没有宝藏的一个门（不是随机打开没宝藏的门，而是有意选择一个没宝藏的门打开），这个时候另一个剩下的门后面有宝藏的概率是 2/3，但是很多人会错误地认为这个门和最初选择的门有宝藏的概率都是 1/2。

整体评价

这本书其他地方倒是没有严重的事实错误，除了之前说的太浅的问题，其他方面倒是一本中规中矩的书。翻译上有几处不似人言的机翻，但是整体还好。

数据密集型应用设计

整体印象

这本书被誉为分布式系统和数据系统领域的“圣经”之一，内容覆盖了数据库、缓存、消息队列、分布式一致性等核心主题。本书最大的优点在于作者对原理的讲解细致且深入——这种深入不是实现细节的堆砌，也不是停留在使用层面，而是从多个角度剖析分布式技术，充分阐述不同技术之间的联系与区别，深入讲解设计思路、权衡取舍和业界常见的设计模式。全书案例丰富，理论联系实际，且尽量避免偏袒某一种技术。

阅读过程

2022 年在知乎的一个计算机书籍推荐帖下看到很多人推荐这本书，好奇之下便买来，但直到 2023 年才开始阅读。前四章看得磕磕绊绊；后来为了应付面试，硬着头皮读复制、分片、事务、墙上时钟等章节，读得非常痛苦，始终不得其精髓。当时感觉这本书就是在不停地罗列名词和技术，似乎很多内容都是浅尝辄止，评论区也有不少人说看不懂，于是便搁置了。最近几个月重新拾起这本书，很多当初读不懂的章节忽然就豁然开朗了，发现这确实是一本讲故事讲得很好的书，有很多章节会让你直呼“原来是这样”。精彩的章节有很多：例如第 3 章用渐进式的例子讲解 LSM-Tree 和 B+ 树索引的设计思路；第 5、6、7、9 章讲解的复制、分区、事务、一致性和共识更是全书的核心，对分布式系统中构建应用的挑战和解决方案介绍得非常深入；即使是批处理、流处理和数据系统未来这类章节，也有非常巧妙的视角来审视现有技术。

阅读门槛

这本书确实不能说易读，毕竟涵盖了大量技术，不可能每一种都详尽介绍，只能介绍其核心思想，否则全书恐怕要写到 5000 页以上。这要求读者具备一定的前置知识，但这只是书本身定位和篇幅的取舍，我不认为算是缺点。另外，一个小小的阅读障碍是章节标题的层级不太清晰，需要主动留意章节之间的从属关系。

Java 编程思想 - LLM 时代读书反思

阅读缘起

这本书很早之前准备学习 Java 的时候就已经买了，当时是指望靠它入门 Java 的。它和 C++ Primer 类似，都是非常厚重的大字典书。因为太厚，买来简单看了一两百页就放下了。最近刷到一道 Java 基础面试题，问的是容器迭代时为什么不能直接用集合自身的删除方法修改结构。底下一条评论嘲讽道，AI vibe coding 时代了，这东西已经没有用了。但是从我的认知来讲，如果要进行专业监督下的 Vibe Coding，或者 Review 代码，起码要知道有这类问题。忽然想起来自己也有一本过时的大厚书，虽然知道 ROI 极低，但一直惦记，最后还是拿来读了。

阅读收获

因为这本书本来就偏字典型，所以我没有深入追究其中的每个细节。比如第 11~13 章讲容器、异常处理、字符串，我基本上就是翻翻就过去了。没想到的是，读完之后还是有一些意外收获：比如第 10 章内部类、第 15 章泛型，甚至第 21 章并发，也让我重新回忆了一下如何用条件变量或者信号量去实现一个 CountDownLatch 和 CyclicBarrier。整体来讲，收获比我想象中多一些，但也确实和我预想的一样，信噪比很低。书里有大量晦涩冗长的设计模式和用法用例，整体读下来，感受也很接近我对 Java 语法的印象：冗长、复杂。同时书本身也有点旧了，主要停留在 Java 5/6 时代的语言视角。

AI 时代的读书问题

我不禁思考另一个问题：AI 时代下，程序员读书还有什么意义？尤其是 AI 诞生之后，大量围绕具体工具和热点应用的图书，例如 Prompt 怎么写、如何构建 RAG、如何搭建 Agent，在 AI 应用高速发展的时代，不到两三个月就可能过期，显得格外可笑。也有一群悲观的程序员，在小红书上晒出自己的书架，里面也有很多《鸟哥的 Linux 私房菜》、Thinking in Java 这种大厚书，然后感叹这些书已经没用，程序员的黄金时代已经过去。那么 AI 时代什么书应该被淘汰？什么书还值得坚持看？看书的理由是什么？如果不看书，又该如何学习一门新知识、新技术？

学习方式与书的价值

从我的理解看，AI 时代学习新技术更友好、更高效的方式，应该是先看大纲式的小册子、讲义、官方文档（如果信噪比足够高，同时足够有趣），或者做一个中小型项目，再结合 AI 去深挖理解。如果确定要深入了解一个领域，再用 AI 辅助阅读底层原理书，或者核心框架源码。这样信息密度更高，知识更新更加及时，反馈也更及时，学习经常有一种初学者的爽感。知识和 AI 话术双重刺激，甚至会让人忍不住兴奋地跳起来。

这样看就很清晰了：书的价值不在于单纯提供信息，而在于提供稳定结构、抽象框架和长期不过时的判断力。

  • 信息密度低、以罗列 API 和用法为主的大厚书，作为入门材料的性价比会明显下降。
  • 一些依赖具体框架的书，例如教你搭建 Agent、教你使用 PyTorch 的框架书，也会非常快地过时。
  • 一些基础的书，比如《计算机网络：自顶向下方法》、JVM 原理，这类书迭代相对较慢，深入底层原理，依然会保持部分活力。

Vibe Coding 的另一面

AI 不仅仅重塑了知识学习，也改变了工具使用甚至编码本身。我尝试纯 vibe coding 了一个采集、回测、交易 All in One 的交易系统 Ganja，还有一个面向金融场景的数据库评测 benchmark，完成之后我甚至不知道目录结构。感觉 Vibe Coding 时代确实人人都能梦想成真，但最终做出来的东西也确实会有很多 bug。AI 会缺乏大局观，写出非常差的系统设计，还会写出非常多防御性冗余代码（不出一天代码就能膨胀到上万行）。如果每次功能设计和迭代，都能事前做好功能设计、方案设计，事后做好测试和 AI review，甚至人工 review，还是可以做出来非常好的产品的。不过这些事情后续 Ganja 单独出个博客来讲。
