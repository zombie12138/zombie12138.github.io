Butterfly 主题使用 | 重塑博客分类和标签

发表于2025-12-13|更新于2026-04-25|Collections
|总字数:3.9k|阅读时长:13分钟|浏览量:

目前的博客依然使用的是 NexT, 其风格是极简纯粹, 安装之后几乎不用配置; 但是功能太少, 依然是默认的使用逻辑. 这种 “开箱即用” 的便利在内容积累到一定程度后, 反而成了桎梏. 最大的问题在于信息密度的扁平化. 在默认的 NexT 时间流视图下, 一篇耗时两周写成的深入的源码剖析+实验性能分析和一篇耗时 5 分钟记录的 Docker 环境配置占据了同等的视觉版面. 这导致高质量的思考被淹没在大量的实验流水账中, 读者无法在第一时间感知到博客的核心价值.

为了优化读者体验, 强调重点内容, 优化检索难度, 本文将重构整个博客的 categories 和 tags; 并且更换主题为 Butterfly.

核心重构需求

为了解决上述问题, 结合 AI 提供的优化思路, 我对新版博客的主题选型和功能定制提出了以下硬性指标:

主页布局的 “CMS 化” (非单纯 Timeline)

我不希望主页仅仅是一个按时间倒序的列表. 我需要主页具备 “门户” 属性, 能够进行分栏展示. 例如顶部导航栏可以设置不同的专项, 精品文章; 首页新增磁铁和置顶文章.

需求点: 主题需要支持自定义首页 Widget 或 Layout, 允许我像搭积木一样分配屏幕空间.

重新定义分类和 Tag

之前是只有 Tags, 没有 Categories, 为了支持上面 CMS 化, 需要找出更好的种类划分, Tag 划分.

审美风格: Smart Casual (精致商务)

我需要太二次元的 “花哨” (排除了一些过度魔改的动漫主题), 也不想要太学术的"枯燥" (当前的 NexT).

需求点: 风格需要是现代、简洁但有质感的. 配色方案应以深色/冷色调为主, 配合适当的留白, 呈现出一种"技术极客"的专业感, 类似于优秀的商业技术文档 (如 Vercel 或 Stripe 的文档风格).

主题技术选型

主题名称 核心设计风格 适合内容类型 你的需求匹配度 (主页分栏/多维分类) 复杂度 (折腾成本)
Butterfly 功能怪兽，视觉华丽 综合型，生活+技术 ⭐⭐⭐⭐ (通过配置置顶和卡片可实现) 高
Stellar Wiki/Notion风，文档化 笔记、教程、知识库 ⭐⭐⭐⭐⭐ (独有的 Wiki 系统完美分离专栏) 中
Volantis CMS/门户风，模块化 资讯、技术文档、大量文章 ⭐⭐⭐⭐⭐ (天生支持主页自定义分栏) 极高
Icarus 经典三栏，IT博客风 纯技术博客、Snippet ⭐⭐⭐ (经典但略显古板，布局较固定) 中
Fluid Material Design，优雅 散文、轻技术、摄影 ⭐⭐⭐ (美观，但主页自定义能力弱于前几个) 低
NexT 极简/学术，黑白灰 论文复现、严肃技术 ⭐⭐ (原生不支持复杂主页，需魔改) 低
Async 现代极简，轻量级 个人随笔、简单记录 ⭐ (太简单，很难撑起庞大的分类体系) 极低

我最终选择功能最强大且热门的 Butterfly 主题.

Tags, 类目改造

我一开始也走了一些弯路，例如这里失败的两个维度划分

失败的两个划分

核心技术栈 (categories)

  • Language/C++
  • Language/Java
  • Language/Python
  • Networking
  • Systems：OS/并发/内存/调度等偏系统方向，强调机制与正确性，例如：内存一致性、调度、COW、同步原语、NUMA 语义
  • Compiler：编译/链接/加载/ELF/ABI/工具链/二进制
  • Performance：测量与优化，例如 benchmark 方法学、吞吐/延迟、CPU 利用率、perf、火焰图
  • AI
  • Devops：Docker, 环境部署

深度、体裁

  • DeepDive：源码/机制/实验闭环
  • TroubleShooting：以故障现象→定位→修复为主
  • Benchmark：以测量方法、指标、对照实验为主
  • Note：操作记录、清单、备忘
  • PaperReading：论文、文章复述、评论、复现
  • Survey：资料汇总与对比，强调覆盖与分类

问题

但是这样在实操中有一些问题，比如我的电脑被黑、排查修复 Hexo Bug、MCU 库探究，都会落入 System 或者 Devops 这样宽泛的内容中。这种极度不均匀的原因是而是现在的 category 在做三件互相冲突的事：

现在一个 category 被迫同时回答：

  • 技术对象是什么（OS / 编译器 / 网络 / MCU）
  • 文章在干什么（排障 / 调研 / 修复 / 实验）
  • 文章抽象层级多高（机制 / 工程 / 使用）

同时深度、体裁也是杂糅在一起的

划分原则

首先需要定义统一、有定量可以执行的标准，要求：

  • categories 最核心稳定
  • categories 和 单个 tags 职责单一，展示独立的不同维度
  • categories 或 单个 tags 内部划分均匀
  • 在满足上述要求的同时 tags 的维度、单个 tags 中的种类尽量少

最终划分

虽然看起来完全没有技术含量，但是是我和多个 AI 多轮讨论、举例 Case 解决、AI 批量划分结果然后再讨论，最终解决了很多边界条件和模糊划分最终得到的一套可操作可量化的分类法。比如下面 Intent 层级原来是 Form 体裁（DeepDive、TuoubleShooting、Note、Survey）和抽象层级（Theory、Desion、Implementation、Operation）合并得到，原因就是两者存在太多的重合（Cos 值太大）。

核心技术栈（Categories）

  • Kernel&OS：包含：Linux Kernel、进程调度 (CFS)、虚拟内存 (VMA)、文件系统 (Ext4/VFS)、并发原语 (Mutex/Atomic)、虚拟化技术 (KVM)。
  • Architecture：CPU 架构、GPU/异构计算、Memory Hierarchy、AI 推理优化（侧重性能和软硬件协同）。
  • DistributedSystem：一致性、调度、分布式存储、K8s 控制面
  • Networking：网络协议与通信机制（TCP/IP、HTTP、QUIC、RDMA、RPC、SDN）
  • PL & Compiler：编程语言理论与工具链（语言特性、运行时、内存安全、LLVM、链接加载、ABI）
  • AI：包含机器学习、深度学习、RAG、模型训练/推理。（侧重算法、应用）
  • Database：数据库、缓存、消息队列、存储引擎（Redis, MySQL, ClickHouse, Kafka, LSM-Tree）。
  • Embedded：嵌入式与端侧开发（IoT、边缘计算、软硬件协同、STM32/ESP32、驱动开发）
  • Engineer：具体工程对象的构建、调试、修复与集成（开源项目、工具链、业务系统实例），上述方案均不满足情况下的逃生通道
  • Collections：一些收藏，比如收藏的博客、名言、博客改造记录；如果一些调研完全确认也没有合适分类，也可以放到这里，作为一种逃生通道

意图（Intent）

  • Survey：生态调研、概念对比、读论文综述。（侧重 Breadth）
  • Theory：纯理论研究，例如 HashMap 算法理论性能、性能模型、算法和模型深度解析。
  • SourceCode：深入系统分析源码
  • Optimization：性能调优过程，最好是实际原理，侧重具体样例分析。
  • Benchmark：压测、定量对比、实验验证、数据报告。（侧重 Data）
  • TroubleShooting：故障排查全过程（现象 → 定位 → 修复 → 复盘）
  • Design：系统设计、架构决策、模块划分、Trade-off 分析、系统分层
  • Reading：论文、博客、书籍阅读。需要阅读具体的内容，一般不能和 Note 共存。
  • Note：日常笔记，备忘录，环境配置清单、命令速查、Cheatsheet。例如记录或者转述原理；记录安装过程等。

技术限定（Tech / Tags）

  • 原则：只能使用具体技术名词，严禁 使用 Category 同粒度的同义词（如 tag 中不应出现 “Network”, “System”）。
  • 编程语言：Rust / Go / C++ / Python / TypeScript
  • 组件与工具：ClickHouse / TDengine / Prometheus / Frigate / LLVM
  • 协议与标准：RoCE / gRPC / Raft / MQTT / io_uring
  • 具体硬件：NUC / RaspberryPi / STM32F103

系列/项目

  • Project/XXX（必须含前缀）：项目上下文（当前仅有 Project/Abyss, Project/Blog, Project/HomeLab）
  • Serial/XXX（必须含前缀）：持续追踪的专题（Serial/JVM）

可选（Optinal）

  • Encrypted：加密文章
  • Featured：精品文章，不超过 20%。

组合约束（强约束）

  • Category：严格 1 个（根据文章核心归属定“户口”）
  • Intent：0-2 个
  • Tech：0–4 个（粒度必须比 Category 细）
  • Optional：根据实际情况

问题依然存在

尽管和 AI 讨论多轮，部分 AI 依然存在意见，主要是 Categories 没法去做互斥。由于 Hexo 的限制，当前一个文章只能属于一个类目，并且我也是认可这个限制，但是事实是可能有多个核心技术栈。一个情况就是 AI Inference 的调优，比如会涉及 AI 和 Architecture 两个部分；分布式+ 的情况也是非常常见，当前标签体系几乎无法解决。之前还想 Intent 做成互斥的，但是也被 AI 辩驳了，Intent 的互斥性更弱，比如 Desion + Survey 的深度调研；Theory + Benchmark 的理论研究 + 实践。这种例子数不胜数，发现，和专注一解决单一事情的论文、发布前就已经定义分类的论坛不同，在一篇复杂的解决真实问题的博客中，类目的互斥是非常难以做到的。

最后分类中不得不引入曾经删除过的 Engineer 类目，但是对于某些笨蛋 AI 似乎不是什么好选择，即便 prompt 中再加强调，AI 依然忍不住偷懒往里面加，最终不得不人工 Review，手动分配兜底的 Engineer。

总结
