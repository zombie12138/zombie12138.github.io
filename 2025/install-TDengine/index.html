安装 TDengine

发表于 2025-02-04 更新于 2025-08-05

最近想要做点量化交易的回测分析什么的, 首先第一就是数据库的选择, 最终选择了 TDengine.

环境: rk3399 2 Cortex-A72 + 4 Cortex-A53; 2 GB RAM

Clickhouse 从 pull 到 give up

官方网站 提供了几种安装方式, Linux 包, 源码编译, Docker. 本着能 Docker 就 Docker 的想法, 最终选择了 Docker 来配置.

Dockerhub 查看说明, 提示:

The arm64 image requires support for the ARMv8.2-A architecture⁠ and additionally the Load-Acquire RCpc register. The register is optional in version ARMv8.2-A and mandatory in ARMv8.3-A⁠. Supported in Graviton >=2, Azure and GCP instances. Examples for unsupported devices are Raspberry Pi 4 (ARMv8.0-A) and Jetson AGX Xavier/Orin (ARMv8.2-A).

但是我贫瘠的 CPU 不太可能支持这个, 尝试使用镜像, 果然不可以

对于这种情况官方给的方案是 create your own image with a binary compiled for old arm cpus. 参考 ISSUE 来进行处理

本来想要自己来编译出来一个 clickhouse, 但是这个光 clone 就下载了很多很多个 GB 的各种源码, 如果编译的话目测几个小时起步, 还需要交叉编译, 可能会遇到各种问题

于是就放弃了, deepseek 给的结果还蛮准确的, 仅仅是作为 db 来存储 + 压缩, 没有必要非要安装 clickhouse.

DuckDB 测试

通过一些 文章 我们大致了解到, DuckDB 是一个 In-Process OLAP 的 DB.

  • 是一个类似于 sqlite 的嵌入式数据库, 但是是 ap 版本, 小而美;
  • DuckDB 使用 vector format 而不是 tuple, 使用 data + selection, 来访问数据, 原生地支持 struct 和 list;
  • pull-based 的执行引擎, volcano 模型;
  • single-file block-based 的存储格式, 通过 header 来实现了 ACID, 每个 block 是 4K 的大小
  • 每张表存储的时候会先水平切分成不同的 row group (每个大约 120K~ rows), 每个 row group 内部的存储是列存的格式; 每个 row group 内部列存储使用压缩, 没有使用 zstd, snappy 这种重量级压缩, 使用轻量级压缩 (目测压缩效率堪忧)
  • 支持 extension
  • 支持 wasm

TDengine

根据 TDengine 白皮书:

  • 为了充分利用时序的特点, 一个数据采集点一个表; 为了解决采集点众多的情况, 使用超级表 SuperTable 来作为同一类型的数据采集点, 同时可以给每一个表设置标签;
  • 可以基于标签对 stable 来进行查询;
  • 内存中 SkipList 作为索引结构, 使用行式存储; 外存中使用分块的列式存储增加压缩能力;
  • 写入流程如下图
  • 单个表不会跨节点存储
  • 每个 dnode 表示一个实例, 包含逻辑虚拟节点 vnode 和 0 到多个的管理节点元数据的 mnode;
  • 不同 dnode 会进行冗余, Master slave 使用同步复制, 全部成功才会返回写入成功.

总结

我们的要求是存储大量的时序的交易数据, 要求是尽可能多节约空间, 用于回测分析.

根据 Deepseek 的建议, 测试了三个 OLAP 的数据库.

  • ClickHouse: 功能强大, 社区完善, 但是安装最为复杂, 对于 Arm 的支持有限, 编译存在计算资源不足的问题, 并且需要交叉编译, 存在未知风险;
  • DuckDB: 嵌入式 OLAP 数据库, 类似 SQLite, 但是压缩性能较差;
  • TDengine: 列式存储的 TSDB, 压缩性能良好, 生态 (学习资料基本上只有官方文档) 有待进一步完善.

最终在 King3399 的嵌入式 ARM 主机安装了 TDengine 的 Docker 服务器

调研的时候意外发现了一个 DB 的 ranking 网站, 就是不知道是什么来做的排名, 似乎是按人气多一些. 想过用 InfluxDB 和 timescaledb 来做, 但是看了 TDengine 的 官方性能对比报告, 并且到手之后比较好的满足了我的需求 (只有一个就是 节约磁盘), 就没有尝试.

附录 - DeepSeek 的呓语

好的，我现在需要帮用户选择一个适合存储高频交易数据的数据库。用户的需求包括：每秒存储交易价格、深度等信息，用于回测分析，需要数据库易于用Python和C++读取，高效压缩列存储，支持时序数据，并在ARMv8a架构上运行良好，节省内存。

首先，用户的需求很明确，主要是高频金融数据的存储和后续分析。这类数据具有时间序列特性，所以时序数据库是首选。同时，用户提到列存储和高效压缩，这通常与分析型数据库相关，比如列式存储的数据库能更好地处理大量数据查询和聚合操作，压缩也能节省存储空间。

接下来，支持ARMv8a架构和节省内存，这点需要注意。有些数据库可能对ARM的支持不够好，或者在某些架构上性能不佳。比如，InfluxDB虽然流行，但社区版可能在集群或高可用方面有限制，且ARM支持可能需要确认。TimescaleDB是基于PostgreSQL的，PostgreSQL本身对ARM支持不错，但TimescaleDB作为扩展，可能需要检查是否有ARM的优化版本。

列存储方面，ClickHouse是一个很强的候选，它专为OLAP设计，列式存储，压缩效率高，查询速度快。但ClickHouse在ARM上的支持如何呢？需要确认是否有官方支持或者社区版本在ARM上运行良好。另外，用户提到节省内存，ClickHouse在处理大数据时可能占用较多内存，但高效的压缩可能抵消部分内存消耗。

Apache Parquet是文件格式，如果用户想用文件系统管理，可能需要结合其他工具，比如DuckDB。DuckDB支持列存储，嵌入式，适合分析，且支持ARM，内存占用低。这可能是一个轻量级的选择，特别是如果用户不需要复杂的数据库服务，而是直接处理文件的话。

Prometheus虽然时序数据库，但更适用于监控场景，可能不适合高频金融数据的复杂查询。TDengine是专为时序设计的国产数据库，资源效率高，ARM支持好，适合物联网和金融时序数据，这也是一个候选。

Cassandra是宽列存储，适合写多读少，但用户可能需要更高效的时序查询，可能不如专门的时序数据库合适。

总结候选名单：

  • TimescaleDB（基于PostgreSQL，时序扩展，ARM支持）
  • ClickHouse（列存储，高效压缩，需确认ARM支持）
  • TDengine（专为时序，ARM支持，资源高效）
  • DuckDB（嵌入式，列存储，适合分析，ARM支持）
  • InfluxDB（时序，但社区版功能有限，ARM可能支持）

需要进一步考虑的点：

  1. 数据写入速度：每秒的数据量很大，数据库的写入性能是否足够。
  2. 查询需求：回测分析可能需要复杂查询和聚合，列存储更适合。
  3. 系统资源：ARM架构下内存和CPU的限制，需要选择轻量级或优化好的数据库。
  4. 易用性：Python和C++的库支持是否良好。

比如，ClickHouse有Python的客户端库，C++也可以直接使用，但ARM支持可能需要查证。TDengine官方文档显示支持ARM，且有Python和C/C++的接口，可能更适合。DuckDB作为嵌入式库，无需服务器，适合单机应用，节省资源，但管理大量数据文件可能不如数据库方便。

权衡下来，如果用户需要高性能的列存储和压缩，且确认ClickHouse在ARMv8a上的支持，可以选择它。否则，TDengine或DuckDB可能是更稳妥的选择，特别是在资源受限的环境下。TimescaleDB也是一个可靠选项，但可能在列存储和压缩方面不如ClickHouse或TDengine高效。

根据你的需求，高频金融时序数据存储需要兼顾写入性能、压缩效率、时序优化和ARM架构适配，以下是经过筛选的推荐方案：

首推组合：TDengine + DuckDB

  1. 核心存储层：TDengine
  • 时序优化：专为高频时序数据设计，数据分区和聚合查询效率远超通用数据库
  • ARM性能：官方原生支持ARMv8架构，在树莓派/飞腾等平台有生产级部署案例
  • 压缩黑科技：列存压缩率可达5%~10%（如1GB原始数据压缩至50MB）
  • 写入吞吐：单节点可支撑百万点/秒写入，完美匹配交易所数据频率
  • 内存控制：数据块内存管理机制避免OOM，适合长期运行
  • 开发友好：提供Python/C/C++/Go等驱动，支持REST API
  1. 分析加速层：DuckDB
  • 嵌入式列存引擎：将TDengine原始数据按日期分片导出为Parquet
  • 内存分析：在Python中通过dtype优化内存布局，比pandas节省50%内存
  • C++协同：通过Arrow格式与C++策略引擎无缝对接
  • ARM向量化：利用NEON指令集加速回测指标计算
