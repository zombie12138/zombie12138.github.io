从 Claude Max 被封到自建 LLM 网关：聚合、审计和 Agent 上下文存储

发表于2026-07-03|更新于2026-07-10|Engineer
|总字数:7.3k|阅读时长:24分钟|浏览量:

之前很长一段时间，我的 AI coding 工具都非常固定，可以说生活在象牙塔里，几乎所有 vibecoding 全都在 Claude Code 中完成。Claude Max x5 也从来没有过 Token 用不完的危机，相反需要每天定目标消耗 Token，东西做错了就直接重做。虽然有 Cursor Pro，但是一直吃灰，当 vscode/网页问答用。听说 Codex Review 很厉害、GLM/Deepseek 又出了 1M 上下文的新模型也从来不关注。

直到账号突然被封禁。

本文由 AI 辅助写作，经作者审核修订后发布，文章 AI 含量：50%；代码 AI 含量 80%。

我的 vibe coding 模式

被封之前，我几乎没有过多尝试其他工具。虽然之前从 chatgpt 横空出世震惊世界开始，尝试过 copilot、cursor、trae，但是直到 2025 年底我才开始充值并且使用 cursor，深深地被 vibecoding 的发展所震惊。但是由于 token 消耗太快，便购买 claude pro，过了两天就买了 Claude Max，从此一发不可收拾。

我买的是 Claude Max x5，日常 coding 直接上最好的模型（截至封号前使用的是 Opus 4.8），完全不需要算 token。所有的 vibecoding 基本全在 Claude Code（下文称 cc）完成，甚至个人笔记之前想过搭建 RAG 来管理 AI 对话和各种笔记，但是也被 cc 暴力接管，博客文章也经常让 Claude Code 辅助撰写。

因为 Claude 对于账号管控非常严格，因此我是通过我的 Google 账号进行订阅（多交 25 USD 的谷歌税）。并且我一直在固定的服务器上工作，日常就是 WindowsTerminal + OpenSSH + Tmux + CC，走的是固定的代理节点，日常会话中英文混杂，甚至最近几个月因此时区也改成了美西时区。

那个时候我几乎不关心每次上下文有多长，也不关心输入 token 到底多少钱，相反我焦虑的是订阅 Token 花不完。Max 订阅把这件事包掉了，我只需要关心模型好不好用，并且经常浪费 token 训斥 AI。那是我至今也非常怀念的一段日子，只要一次交够钱，就能肆无忌惮地把一个项目不停的推倒重来。

封禁和申诉

其实最近已经是越来越严格了，在互联网和身边也有人控诉，MAX 充值一两周就封号，CC 源码泄露时候也有人深挖其风控收集的信息，及其细致严苛。但是我还算比较庆幸，我的账号虽然直接用的代理，但是整个账号一个地点登录，固定 IP，还算毕竟稳定。直到 2026 年 6 月 30 日的下午，我发现账户需要重新登录，由于这个也曾经出现过，我也没有在意，就出去玩了。晚上回来发现依然无法登录，登录网页 <claude.ai> 发现已经是免费用户了。查看邮件，收到了一封邮件：

Claude Max 被封之后，我第一反应就是找原因，我第一反应是最近网络波动较多，可能代理行为、网络重试、最近激增的中文输入这些特征导致的，后面又怀疑是不是最近做的一些抢号脚本导致违反用户协议。开始我还挺慌，也抱有一丝幻想，没有查攻略就找 Gemini 帮我写申诉邮件，解释自己是正常 coding 使用，不是薅羊毛，也不是批量滥用。给 support@anthropic.com/usersafety@mail.anthropic.com 发送申诉邮件，但是收到的只有机器人自动回复：

后来刷小红书才知道可能是由于 Ali 蒸馏的原因，亦或是 AI 安全敏感的原因，最近普遍进行了封号，并且一些直接租海外服务器 SSH 使用的人、甚至留学生都被封号。我则可能最初已经暴露，浏览器/操作系统的特征信息、大量的中文对话，只不过这次风控阈值调低才真的暴露。看到有人说有专门的申诉入口，support 里面找 fin AI，告诉它没有申诉入口就会给我一份专门的申诉表单入口。于是又去填表，重新解释一遍，但是回复给我的是只有自动模板。

更离谱的是，Cursor 那边也出问题了。之前一直不用它，现在突然想拿来救急，但是也只能使用大陆地区的模型。我是直接 Alipay 支付的，通过本地 TUN 模式代理 + HTTP1.1 协议使用的，猜测也因为 Anthropic 的要求或者一些敏感原因。

Alternative

AI 一旦挂了，整个工作会变得非常痛苦，现在只有一个残血 cursor 可以用，但是我更加习惯 CLI 的 coding agent，当务之急是找到替代。

GLM

首先是尝试了 GLM-5.2，这个应该是力大飞砖的最强国产编程模型。

这里不得不吐槽下 GLM 的官网，聊天、Coding Plan 套餐、个人中心、文档分散在很多不同的入口，我最终直接用 API 验证才知道，普通余额可以直接使用，Coding Plan 也可以。并且，不同的 token 用量，缓存命中率这些完全看不到，甚至连消费情况也是按月统计。

配置好了 CC+GLM5.2 之后，但是按量付费让我第一次认真意识到 token 问题。之前 Max 订阅下没有感觉，换成 GLM API 后，我测试了一个复杂的性能问题，它直接干了半个小时烧了 49 CNY，但只用完了 200k/1M 的上下文。后续日常使用小半天就能烧掉 100 CNY。按照这样算，我之前一天用量得 500 元才能支撑。但是不得不说，GLM 用起来体验和智商还是非常接近 opus 4.8 最新模型的，但是 opus 大概率是残血蒸馏的，但是 GLM 则是靠力大飞砖，高昂的推理成本就是模型 token 天价的原因。我也试过去抢智谱的 Coding Plan，但是没抢到，后面就放弃了。

Codex

再后面我买了 OpenAI Pro。Codex 能用，首先就是非常 节约 token，体感上能比 CC pro 多做一倍的工作，同时出 token 速度也非常快。但是我还是觉得它的工作方式没有 Claude Code 那么贴手，并且最大的问题是 太跟随指令了。我之前要求任何问题有歧义需要停下来咨询我的意见，cc 只在关键位置停下，但是 codex 我一次需求会停下来咨询数次，甚至完全没有影响的、排查 bug 的方案也要停下来咨询我；我之前也要求 cc 每次输入英文并且表述有问题先纠正我，之后再回答问题、工作。但是 cc 很少遵循，反倒是 codex，哪怕我输入中文也会不停地纠正我的表述。

这样我的 codex 一个 pro 订阅覆盖大部分的开发工作；GLM+CC 则是更加强也更加昂贵的补充，或者仅仅指定 plan；Curosr 辅助一些简单任务，做日常 IDE 用。

agent-link-skill: 快速同步 cc 和 codex

但是 CC 和 Codex 的 rules、memory、plugin 乃至 上下文都不共享，交替使用会有很多麻烦。为了解决上下文共享，我写了一个 skill，叫 agent-link-skill。

它做的事情非常简单，就是用 symlink 把 Claude Code 和 Codex 的一些文件连起来。比如项目规则里 Claude Code 读 CLAUDE.md，Codex 读 AGENTS.md，那就让其中一个指向另一个，这样两边文件是完全同步的。项目记忆也是类似，让 Codex 的 memory 目录指向 Claude Code 已经整理好的 reviewed memory。这个功能看起来小得不能再小，指令也只是几句话的事情。但是为了让这个过程更加流畅，行为更固定，我还是选择 skill 把它固化起来。

但是做起来比我想象中繁琐。因为 Claude Code 和 Codex 对 rules、skills、memory 的管理不同。对于 rules 这种单一文件非常好解决；memory 则是 claude code 就可能存在多个可能的文件夹；Skill 则是最为复杂。Codex 的 Skill 包含自己的 .system，Claude Code 存在一个插件目录和一个 Skill 目录两个来源。考虑到 codex .system 不应该向 CC 暴露，最终放弃同步整个 SKILL 文件夹，选择逐个同步。后续安装自动触发 Skill 进行同步也比较复杂，需要用户手动配置 hook 定期检查，或者靠全局 rule 来约束提醒。最终选择让 skill 保持简单，仅仅做同步。

更多的问题是脚本和 prompt 的边界问题。写 prompt 的 codex 最初是所有功能全都放脚本，尽管 prompt 已经包含不同的情况处理。首先就是脚本会非常复杂，一个这样的简单的需求，我让 AI 写了 400 行晦涩难懂的代码，完全没法 review。这个甚至 prompt 给个 symlink 的命令行都可以，拉扯多次，最终让 AI 代码保持在了 200 行附近。

至于上下文，这里懒得解析 AI 存储上下文的方式了，这个太不稳定，可能今天存在 json 中，明天又存 db 里，最好的方式就是 compact 之后总结的信息发送给另一个 agent。但是 CC 现在不显示总结的原文，即使 verbose 模式也看不到，我后续是直接靠 LLM 审计的网关截获的记录拷贝复制过去的。

Claude Code 联网搜索

Claude Code 的 WebSearch 依赖 Anthropic 服务端搜索。把 ANTHROPIC_BASE_URL 指到第三方 API / 自建网关之后，自带 WebSearch 就无法使用了，表现就是界面里 Web Search … Did 0 searches。

火山 Agent Plan 送了「豆包搜索」（Skill / MCP，每月约 500 次免费，超出 5 AFP/次）。我实际装过：以 Skill 形式安装，运行时走 CLI 脚本调搜索；但是最大的问题是体感上对英文资料偏弱，结果噪声大，不太能当主力。

LLM API Gateway

初衷

搭建 LLM API 网关的初衷有很多：

  • 第一是规避 CC 的一些最简单直接的监控。据说，Claude Code 客户端本身还有很多采集用户数据的情况，堪比木马。但是我又想享用世界最强 harness 的更新。这里可以做透明代理拦截 Anthropic 的请求，但是这里我想匿名化 BASE_URL 和 model name。
  • 第二是统一 管理 API。不同平台的 key、base_url、模型名、价格都不一样，手动塞进每个客户端非常乱。尤其是以后还可能接 Doubao、Kimi、各种套餐、第三方中转等。
  • 第三是 协议转换。Claude Code 只接受 Anthropic 的协议；还有很多则是 OpenAI、Gemini 等协议。没有网关做双向转换，CC 就无法对接这些上游，或者每个客户端各接一套桥。
  • 第四是 请求审计。我想知道每次请求到底发了什么、模型回了什么、走了哪个上游。更进一步，审计内容也可以拿来做 跨 agent 的上下文迁移。比如 Claude Code 的 compact 现在是黑盒，不给看 compact 内容。虽然也可以导出，但是需要 agent 重新输入到文件中，再让另一个 agent 读取。
  • 最后希望可以进行 更细致的用量统计。

选型

cc-switch 是一个轻量、专业地切换不同供应商的工具，它可以切换 ANTHROPIC_BASE_URL 和 token，但是它在客户端侧，是 改写客户端配置文件，不在请求链路里。它不做服务端聚合，也不做审计。所以我最后还是决定部署一个真正的 API 网关。

调研以后，new-api 比较适合我的场景，它是 one-api 的 fork，使用 go 重写，并且依然在即使维护。它有后台，有渠道管理，有 token 管理，有计费，也能做模型映射。LiteLLM 也很专业，provider 支持更广，但是我这个阶段更需要一个容易部署、后台完整、计费直观的东西。

new-api 自己有日志，但我想要的不只是计费日志。我想看 Claude Code 发出的完整 /v1/messages，以及上游返回的内容。所以又加了一层审计代理。为了方便管理，new-api 层做聚合，审计层仅仅做审计，因此放在网关和客户端之间，对聚合后的内容进行审计。

审计可以采用 Langfuse，但是这个是一个终点，不能转发，只能做旁路。这里使用 Mitmproxy 做审计层实际嵌入链路的工具，仅仅做客户端和 new-api 网关之间的转发，同时 Hook 请求转发到 Langfuse 存储和展示。

审计层

审计层分两部分：audit-proxy 嵌在 CC 和 new-api 之间，负责截获流量；以及负责实际接受到请求存储和展示的：Langfuse。

mitmproxy

mitmproxy 是一款开源、可编程的 HTTP/HTTPS 代理，通过 Python addon 在 request() / response() 等钩子里截获并处理流量。我们用 mitmproxy 做反代，hook 请求转发到 Langfuse 做存储。

最终流量从 CC 发过来之后经 mitmproxy 转发到 new-api，最终发给 LLM 厂商。它在 request() hook 里解析 /v1/messages 的请求体，并且记录到 flow；在 response() hook 里解析 Anthropic JSON 或 SSE 流式响应，并且记录到 Langfuse 中。

需要做的只有写大概一两百行的解析和存储，写个 Docker file 和 compose yml，扔 docker 里面即可。

Langfuse

Langfuse 自托管是一组服务：

  • Langfuse (web)：包含 Web UI 和 HTTP Ingestion API，可以登录查看审计的请求详情，也可以做训练数据打分和管理。audit-proxy 里的 Langfuse SDK 把 trace/event POST 到这里；web 接收 ingestion 请求后把任务推送到队列，渲染页面时则从 Postgres、ClickHouse 等读取。
  • worker (langfuse-worker)：后台消费者，不对外暴露端口。从 Redis 队列（BullMQ）取出 ingestion 任务，解析、规范化后批量写入——观测数据进 ClickHouse，元数据和关联关系进 Postgres，大块 raw event / media 进 MinIO。UI 里的 trace 详情，多半要等 worker 异步落库后才能查到。
  • Postgres：关系型库，存元数据和配置（组织、项目、用户、API Key、trace/generation 索引、会话与权限等）。Langfuse web 查列表、做登录鉴权，主要依赖 Postgres。
  • ClickHouse：列式 OLAP 库，存高吞吐的观测事实表（observations、scores、usage 统计、按时间范围的 trace 查询等）。Langfuse v3 把大部分 trace 明细和聚合查询放在这里，因为单条 generation 可能很大、写入很频繁，Postgres 扛不住。
  • MinIO：S3 兼容对象存储，存大块 blob（raw ingestion events、media 附件、batch export 等）。超过一定体积的 payload 或导出文件放在 MinIO，Postgres / ClickHouse 里只存索引和结构化字段。
  • Redis：内存队列和缓存，不持久化 trace 正文。web 收到 SDK 上报后先把 job 推到 Redis（BullMQ），worker 再消费，用来削峰、解耦——API 快速返回 202，重活交给 worker。

可以看到这个 Langfuse 其实还是非常复杂的，从轻到重给了三种不同的 DB，为了加速写入还有 redis 队列和 worker，但是我们这里用不到这么多复杂的功能（例如训练数据打分和管理相关的），也不会需要特别高性能写入，毕竟只有我一个人使用。因此使用 Langfuse 太重，我本机 48 GB 的内存，一个 Langfuse 就能吃掉 4 GB，但是我用到的功能就只存储和展示，对于性能要求也不大。

这里的 23823 tokens 是请求响应带来的缓存未命中的 token 数，229 是缓存命中的 token 数，和 new-api 一致，并且可以看到请求响应的原文。有了这个就可以看 CC 具体对 API 每个请求的明细；也可以审计和存储具体会话历史；也可以用上下文、compact 信息跨 Agent 进行迁移会话（或者其实简单让 AI 直接写文件也可以解决）。

Input 被截断

故事还要从审计的存储讲起。当前我的 proxy 脚本是进行 20000 Python 字符截断，之后传入 Langfuse SDK，这个对于 1M 上下文本身够用了。但是 Langfuse 本身是存在截断行为的：

但是这只是临时看更多内容。1M token 级别的上下文，不适合直接塞进 Langfuse 的 input 字段里，也不适合靠浏览器页面硬渲染完整 JSON。就算 SDK 放开，后端、数据库、API 和 UI 也可能有其他限制。

Loki 则更加严重，默认超过 256KB 会直接抛弃或者截断，我在转发之前就提前进行了截断，对于 1M 的上下文，用 Langfuse 使用传统数据库来进行存储即使压缩也会有大量的前缀冗余的模式没有被充分挖掘：

  1. 对于 Agent 调用记录，不同请求之间往往共享前缀（但是也可能有 rewind 的例外）
  2. 对于所有会话，可能存在公用的系统提示词、项目规则等前缀
  3. 甚至每次返回的响应，都是下次请求的前缀
  4. 一些 SKILL、Memory、读取到的文件，可能出现在任何会话的任何地方（最难利用）

但是对于公司级别的使用，很多时候可能会需要有一个网关审计记录所有数据，可能也是写入这种传统的 OLAP 或者对象存储，我不禁思考，对于 Agent 时代，是否有更加高效的审计存储的数据库？

KV cache 的思路

笔者在写这篇文章时，还没有对 KV cache 做过系统的学习和研究，仅是初步了解常见优化和概念。但可以发现 agent 审计存储的场景和 KV cache 很接近：block 化、按 hash 寻址、prefix 复用、hot/cold 分层这套机制几乎可以直接搬。但审计的复用条件比 KV cache 更宽松：不要求前缀严格相同，上下文任意位置出现的重复段都能去重。所以仅在前缀复用和 block 存储这一点上，可以借鉴 KV cache 的设计。

最直观的借鉴是 PagedAttention：把上下文切成 block，按 content hash 寻址，一次请求就是一个 block list。光切块去重还不够——agent 多轮请求里大量是前缀复用，新请求的上下文基本是上一次再加几个 turn，可以用一个 prefix tree 管哪些前缀被哪些请求共享，新请求来了沿树命中已存 block，只存后面的 suffix。这样不仅仅可以不同请求之间前缀压缩，也可以解决系统提示词、项目规则等公用前缀的复用。

但 KV cache 的 prefix cache 有个硬限制：前缀必须严格相同才能复用。agent 审计里更常见的是另一种重复——同一段 skill、memory、读到的文件内容，会在上下文任意位置出现，不一定是前缀。这是审计存储可以比 KV cache 更进一步的地方：把 prefix tree 升级成 Merkle DAG，任意位置的重复段都靠 content hash 去重。去重一上来就必须管引用计数——多个请求引用同一 block，refcount 归零才能回收。但是这里的难点在于怎么识别重复模式和匹配模式，例如精准识别出来文中出现的 SKILL 片段、或者多个上下文经常出现的文件片段，比如某一个按需加载的 Memory。这种可以手动规定一些调用 SKILL 的特征来进行识别，但是本身可能收效不会太高，做起来却相当复杂。

对于分布式的存储，还可以把 prefix 重合度高的 session 路由到同一节点。甚至进一步冷热分离等。

现有的一些组件

即使没有审计，也有存储用户请求用于训练的需求，那么必定有公司、社区做过开源组件。

最接近的是 HuggingFace 的 Xet。它就是 CDC + content-addressed + Merkle 的生产实现：文件按 ~64KB 切变长 chunk（rolling hash 定边界）。每个 chunk 算 BLAKE3 hash 存进内容寻址存储（Content-Addressable Storage），chunk hash 再算一层 Merkle hash 做文件完整性校验。上传时先查本地 cache 再查全局 CAS，重复 chunk 直接跳过，chunk 聚合成 64MB 的 xorb 作为传输单元。现在承载 21PB 数据集（Parquet 就占 4PB+），协议开源。

算法层面，CDC 的事实标准是 FastCDC（Gear rolling hash + normalized chunking），开源实现有 fastcdc-rs（Rust）和 Plakar 的 go-cdc-chunkers。Parquet 那边 pyarrow 也加了 use_content_defined_chunking=True，写 Parquet 时按内容定 data page 边界，配合 Xet 这种 CAS 做 dedup——HuggingFace 还专门出了个 dataset-dedupe-estimator 测 dedup 率。

底层文件系统方面，在很早就有 AI-native 的文件系统，DeepSeek 开源了 3FS，MIT 协议。专门优化随机读（training data loading + KV cache lookup），故意不做传统 read cache / prefetch，因为 AI workload 随机访问 cache 命中率低。180 节点 6.6 TiB/s 聚合读吞吐，生产跑 V3/R1 训练 + 推理。它偏重——需要 RDMA + 大量 SSD。

网络协议

更进一步，Agent 每一次和 LLM 的交互都需要发送全量的上下文，1M token 上下文可能就是 4-6MB 的 JSON。我的直觉是这事一定能做 hash + delta：客户端只发新 chunk 的 hash 清单 + 新 block bytes，服务端按清单拼上下文，缺的再让客户端补。

查了一下已经有了，OpenAI Responses API：previous_response_id 让下一 turn 只发新 input，服务端按 ID 取回上次 state 拼上下文。还上了 WebSocket 传输，连接级 in-memory state cache，官方博客原话就是「keep a persistent connection and cache state, rather than sending the full conversation history for each follow-up」。

不过这部分成本比例太低，同时切换协议历史包袱太重，并且需要额外存储原文和原文哈希，或者原文哈希到 kvcache 的映射，大部分 API 还是发送全量上下文。更多的工作依旧是 kvcache 计算和存储相关的。

总结

Claude Max 被封最开始只是一个账号事故。被封之后反而坦然了，最新的 LLM 都不会差太多。后面也是搭建了一套用量统计、审计、聚合的网关。最后一个意料之外的收获：从 Langfuse 截断这个存储问题出发，简单让 AI 调研了下，才发现 LLM/Agent 的 infra 其实除了计算、推理、训练、存储这些最热门的问题，也一定有很多其他的问题。不过这两年已经被业界推得很远了。这也印证了：计算机领域只要理论上存在能实现，就一定能实现，甚至大部分已经有人做的非常完善了。

这里我不想谈论政治相关的话题，仅仅谈谈我个人感受角度。虽然被封了，但是我对于 Anthropic 的情感是非常复杂的：一方面我非常感激他能提供这样便宜的订阅方案，廉价的 Token（仅针对订阅），世界顶级模型和 Harness；但是另一方面他十分针对性打压大陆用户，对封号行为感到气愤和羞辱。从前我对于国产模型发展是几乎不太关注，什么千问出个新模型、DeepSeek 1m 上下文模型，GLM 又堪比世界一流模型，现在我才深深感受到这些的重要性。我的消费也是真金白银的投票，希望这些公司能发展更好。
