HomeLab 可观测性建设 | VictoriaMetrics + Loki + Grafana 全栈实践

发表于2026-03-04|更新于2026-04-25|Engineer
|总字数:12.5k|阅读时长:47分钟|浏览量:

最近用了很久的嵌入式板子连上去发现 docker 和 systemctl 都用不了，重启系统直接挂了。不过好在重要的服务都跑在一台新买的 AMD 8745HS 上。不过现在已经累计 40 多个容器跑在单机上，连哪个容器挂了、哪个容器占用多少内存都不知道。虽然可以使用 docker stats 来观测，但是也无法事后查看。哪一天服务崩掉也只能通过服务业务本身通知，但如果服务彻底崩溃，这个过程是完全静默的。决定彻底解决这个问题：搭一套完整的可观测性栈。

本文由 AI 辅助写作，经作者审核修订后发布，文章 AI 含量：70%；代码 AI 含量 80%。

背景

我的 HomeLab 单机规格：32GB RAM、AMD Ryzen 7 8745H (8C16T)、826GB 磁盘。上面跑着 40 多个容器，分布在大量独立的 compose 文件里：clash & yacd、qbittorrent+wireguard（多实例）、Samba、TDengine、frpc、DST 游戏服务器、交易数据爬取、离线任务调度，以及各种杂项。

部署之前的状态：

  • 指标：没有。只能 docker stats 看当前快照，没有历史记录。
  • 日志：每个容器各自写 JSON 文件，Docker 日志轮转，查日志要先 docker ps 找容器名，再 docker logs 翻滚，效率极差。
  • 告警：没有。服务挂了只能靠自己发现。

可观测性四种信号

现代可观测性有四种信号，从"广度"到"深度"递进：Metrics 告诉你趋势，Logs 告诉你细节，Traces 串联请求链路，Profiles 定位代码热点辅助分析性能问题。一个典型的故障排查流程是：Metrics 发现异常 → Logs 定位原因 → 必要时 Trace 追踪跨服务链路。Metric 和 Log 是互补的，不是替代关系。

信号 存储内容 单条大小 查询速度 典型场景
Metrics 预聚合的数字（计数器/直方图/仪表盘） 8 bytes/点 O(1)，毫秒级 每秒多少请求？p99 延迟？错误率？CPU 趋势？
Logs 每条事件的完整文本 几百 B ~ 几 KB/条 全量扫描，慢 3:14pm 那个报错的完整堆栈？具体哪行代码？
Traces 请求在多个服务间的链路和耗时 几 KB/trace 按 traceID 查 这个 API 慢，卡在哪个下游服务？
Profiles 函数级 CPU/内存采样 几十 KB/采样 按时间段查 哪个函数吃了最多 CPU？内存泄漏在哪？

技术选型

监控栈由哪些角色组成

一套完整的监控栈有四层职责：

  • Exporter 层：把各种服务的内部状态暴露为 HTTP /metrics 端点（格式：Prometheus text）
  • Scraper/Agent 层：定期拉取（Pull）或接收推送（Push）指标，转发到存储
  • TSDB（时序数据库）层：高效存储和查询时间序列数据
  • Visualization 层：查询、展示、告警

之所以是四个组件而不是一个程序，是因为每层的关注点不同——Exporter 只需要懂"如何读 cgroup 和各种指标"，TSDB 只需要懂"如何压缩时间序列"，这种分层让组件之间解耦，可以自由组合。

常见的组合也有很多，例如：

  • PLG（Prometheus + Loki + Grafana）：Prometheus 负责指标采集拉取/查询，Loki 负责日志存储与查询，Grafana 统一展示与关联分析，是当前最常见的轻量开源组合之一。
  • ELK（Elasticsearch + Logstash + Kibana）：日志检索和分析能力很强，适合复杂日志分析场景，但整体资源占用更高。

本文采用的是 Prometheus 生态变体：指标侧使用 VictoriaMetrics（PromQL 兼容）替代 Prometheus 内置 TSDB，日志侧使用 Loki，并统一接入 Grafana，下面我们探讨每一层的详细选择。

指标采集层：node_exporter 和 cAdvisor

指标采集层负责暴露服务内部状态，通常使用 Prometheus text 格式，例如一个真实的例子：

  • node_exporter 读取 /proc/stat、/proc/meminfo、/proc/diskstats 等内核接口，暴露主机级指标：CPU 使用率、内存、磁盘 IO、网络流量。它只读不写，直接以 root 或普通用户运行即可。
  • cAdvisor（Container Advisor）读取 /sys/fs/cgroup/<subsystem>/<container_id>/ 下的 cgroup 文件，同时通过 Docker socket 获取容器元数据（名字、标签、镜像），暴露容器级指标：每个容器的 CPU 份额、内存用量、网络 IO。
  • Telegraf——更通用，可以 同时采集主机和容器指标，但配置更复杂，内存占用也更高。

对于 HomeLab 轻量使用的场景，node_exporter 暴露宿主机指标 + cAdvisor 自动拉取容器指标更合适。

存储与查询层：Pull vs Push & 时序数据库选型

Pull vs Push 是第一个核心取舍：

  • Pull（Prometheus 模型）：Scraper 主动 HTTP GET 每个 target 的 /metrics。优点：scraper 掌握采集频率 流量可控；状态即刻感知，target 宕机立刻感知。缺点：target 必须可被 scraper 网络访问。
  • Push（InfluxDB/vmagent 也支持）：target 主动推送数据到接收端。优点：target 在 NAT 后面也能上报；缺点：接收端宕机时 target 需要缓冲，采集频率由 target 控制。

HomeLab 内网环境，Pull 模型完全够用。

VictoriaMetrics 存储原理

VictoriaMetrics 的存储引擎比较接近 LSM Tree，不过对于时序数据进行了特定优化，核心思路是按月分区 + MergeTree 合并 + 列式压缩。

写入流程：数据写入时先进内存表（in-memory part），积累到一定大小后 flush 到磁盘上的 small/ 目录，形成一个 part。后台 merge 线程持续将多个小 parts 合并成更大的 part，最终移入 big/ 目录。这个过程类似 LSM-Tree 的 compaction，但针对时序数据做了大量优化——比如合并时不需要排序（时间戳天然有序），主要工作是重新压缩和构建索引。

磁盘结构：

每个 part 内部是列式存储——时间戳和值分开存放。同一个 time series 的相邻数据点在物理上连续排列，这对压缩极为有利。

压缩算法：VictoriaMetrics 对时间戳和值使用不同的编码策略：

  • 时间戳：采用 Gorilla 编码（Facebook 2015 年论文《Gorilla: A Fast, Scalable, In-Memory Time Series Database》）。核心思想是存储相邻时间戳的差值的差值（delta-of-delta）。对于固定间隔采集（如每 15 秒一个点），delta-of-delta 几乎全是 0，一个时间戳只需要 1-2 bits。
  • 值：使用 XOR + 变长编码。相邻值做 XOR，如果结果为 0（值没变）只需 1 bit；变化不大时只存变化的 bit 位。对于 gauge 类指标（如 CPU 利用率在 0.3~0.4 之间波动），压缩率极高。

实际效果：每个数据点平均只占约 0.7 bytes（时间戳 + 值合计），而原始的 float64 + int64 需要 16 bytes，压缩比约 20 倍。

过期与保留：通过 -retentionPeriod 参数控制（本文配置 90 天）。过期删除的粒度是月分区——到了第 91 天，整个月分区目录直接删除，无需逐条扫描，删除操作是 O(1) 的。

降采样：VictoriaMetrics 开源版不自动降采样。90 天前和今天的数据精度完全一样（15 秒一个点就一直是 15 秒）。降采样是企业版功能（通过 -downsampling.period 配置）。对于 HomeLab 场景，90 天全精度数据大约占 2-3 GB（当前 50 天约 1.2 GB），完全在可接受范围内，不需要降采样。

关于 TDengine：虽然我有一个 TDengine，但是它已经在跑了，复用能节省 RAM，但是复用会耦合两个服务之间的负载和稳定性，且极少现成 Grafana 仪表盘（要从零搭）、Grafana 插件成熟度差、SQL 语法导致 Prometheus 仪表盘全部不可用。节省的那 0.3~0.5 GB 不值得长期维护成本。

指标采集代理与胶水层：vmagent

如果沿用 Prometheus 一体化方案，链路通常是 exporter -> Prometheus -> Grafana，Prometheus 自己就能完成抓取、存储、查询与规则计算，不一定需要额外代理。

本文把指标存储换成 VictoriaMetrics 后，链路变为 exporter -> vmagent -> VictoriaMetrics -> Grafana。这时 vmagent 负责中间“胶水层”职责：主动抓取各个 exporter 的 /metrics、通过 remote_write 转发到 VictoriaMetrics，并在短暂抖动时做缓冲重试。

日志采集与存储层：Promtail+Loki

日志采集器的选择：

  • Promtail：Loki 官方采集器，和 Loki 原生集成，配置最简单；支持 docker_sd_configs，可通过 /var/run/docker.sock 自动发现容器。
  • Fluentd/Fluent Bit：通用采集器，插件生态完善（可对接 Loki），适配面广，但配置与运维复杂度更高。
  • Filebeat：Elastic Beats 家族；虽支持 Elasticsearch/Logstash/Kafka 等输出，但无原生 Loki 输出，接入 PLG 往往需要中转链路，不是轻量场景的首选。
  • Logstash：更偏向日志处理/转发管道（解析、清洗、路由），常和 Elasticsearch 或 OpenSearch 搭配；功能强但相对更重。

日志存储选型：

本方案里的指标链路（node_exporter、cAdvisor、vmagent）本质上就是宿主机集中采集；日志链路（Promtail/Fluent Bit/Filebeat）在工程上既可宿主机采集，也可 Sidecar 部署。

在本文场景下全部选择 宿主机采集，原因很直接：容器规模较大且分布在大量 compose 文件中，Sidecar 模式意味着要改所有 compose 文件并为每个服务额外部署 sidecar。宿主机模式只需额外维护一份监控 compose，零侵入，且所有采集器都可以以 Docker 容器方式运行。

深度解析：

  • node_exporter 通过 /proc/stat 读取 CPU 时间片（user/system/idle/iowait），通过 /proc/meminfo 读取内存分布，通过 /proc/diskstats 读取块设备 IO 计数器，汇聚成 Prometheus 格式的文本端点。
  • cAdvisor 读取 /sys/fs/cgroup/<subsystem>/<container_id>/ 下的 cgroup v2 文件（如 cpu.stat、memory.current），同时调用 Docker socket（/var/run/docker.sock）获取容器名称和标签，将两者合并成带容器元数据的 metrics。
  • vmagent 只做 scrape + remote_write，不存储任何数据，内存占用极小（~50 MB）。相比 Prometheus server（scrape + 存储 + 查询三合一），vmagent 更轻量，且支持多个 remote_write 目标（方便未来扩展）。

深度解析：

  • Docker json-file logging driver：默认 logging driver，每条 stdout/stderr 输出写成一行 JSON：{"log":"...\n","stream":"stdout","time":"2026-01-01T00:00:00Z"}。日志文件路径是 /var/lib/docker/containers/<container_id>/<container_id>-json.log。
  • Promtail docker_sd_configs：通过 Docker socket 自动发现所有运行中的容器，读取 /var/lib/docker/containers/<id>/<id>-json.log，无需手动配置每个容器。容器新启动后 5 秒内自动被发现。
  • relabel_configs：从容器 label（Docker Compose 自动注入的 com.docker.compose.project、com.docker.compose.service）提取 compose_project、compose_service 标签，使得 LogQL 查询可以按服务名精确筛选。

为什么合并在单个 compose

  1. Grafana 是共享的 — Grafana 同时连 VictoriaMetrics 和 Loki，拆成两个 compose 文件要么重复部署两份 Grafana，要么靠外部网络跨 compose 通信。放在一起自动共享网络，depends_on 直接用。
  2. 服务间依赖在同一个 compose 里最简单 — vmagent → victoriametrics、promtail → loki、grafana → victoriametrics + loki，这些依赖关系 depends_on 一行搞定。
  3. 这组监控容器是一个功能单元 — “监控”。要么一起起要么一起停，docker compose up -d / down 一次搞定。

部署

目录结构

准备工作：data 目录权限

在 docker compose up 之前，需要先建好数据目录并修正权限——Grafana 和 Loki 以非 root UID 运行，若目录 owner 是 root，它们会报权限不足立刻退出：

VictoriaMetrics 和 Promtail 写数据目录时会自行处理权限（配合 CHOWN/DAC_OVERRIDE cap），无需提前 chown。

安全加固：capabilities

所有容器统一 no-new-privileges: true + cap_drop: [ALL] + 按需 cap_add，只开放最小权限。

no-new-privileges: true 的作用：防止容器内进程通过 SUID/SGID 二进制提权（setuid bit 不再生效），即使容器内存在 SUID 程序也无法用它切换到更高权限。

各 capability 的含义：

Capability 作用 使用服务
CHOWN 修改文件 owner（chown 系统调用） VictoriaMetrics、Loki、Grafana
DAC_OVERRIDE 绕过文件读写权限检查（不管 owner/group） VictoriaMetrics、Promtail
DAC_READ_SEARCH 绕过文件读取和目录遍历权限检查 Promtail（读 Docker 日志文件，owner 是 root）
SETUID / SETGID 切换进程 UID/GID Loki、Grafana（启动时内部降权）
FOWNER 绕过文件 owner 校验（fchown、fchmod 等） Loki、Grafana

cAdvisor 的例外：cAdvisor 必须用 privileged: true，不能用 cap_drop: ALL。cgroup v2 + Docker 29 环境下，读取完整的 cgroup 层级结构需要 root namespace 的访问权限，现有的细粒度 capability 无法覆盖这一需求。关于 Linux 中的权限问题可以后面再单独来讲。

cAdvisor 兼容性：Docker 29 + containerd image store

cAdvisor 在 Docker 29+ 上的兼容性问题可以分成两层：

  1. API 层（先决条件）：Docker 29 要求最低 API 1.44。旧版 cAdvisor（如 v0.49.1，client 1.41）会在初始化阶段直接失败，Docker factory 无法注册（Issue #3749）。
  2. 存储层（路径/驱动适配）：启用 containerd image store（containerd snapshotter）后，容器层元数据路径和驱动语义发生变化（如 overlay2 -> overlayfs）。cAdvisor 旧实现按历史路径解析 rootfs，会触发 failed to identify the read-write layer ID for container，导致容器级 metrics 缺失（Issue #3643, Issue #3860）。

按版本看：

  • v0.49.1：卡在 API 层（1.41 < 1.44），启动阶段即失败（Issue #3749）。
  • v0.52.1 ~ v0.53.0：API 层通过，但存储层不兼容——在 Docker 29 + containerd snapshotter 下，容器级 metrics 全部缺失，仅可见系统 cgroup（Issue #3860）。
  • v0.54.0+（2024-12-02）：PR #3709 改为基于 containerd OCI spec 读取 rootfs 路径，兼容性明显改善。

实践上建议直接使用 v0.55.1（本文已验证可用），并启用 privileged: true + --store_container_labels=true。

非 Docker 日志源接入

Docker SD 自动采集的是容器 stdout/stderr，但有些数据源不走 Docker 日志：容器写文件的日志、宿主机 systemd journal、内核级 execve 事件、Docker 生命周期事件。这些需要单独接入 Promtail。其中 Docker 生命周期事件（docker events）是为了容器告警引入的，采集配置在后文告警章节说明，采集模式和下面的 journal、Tetragon 完全一致：宿主机 systemd service 导出 JSON 文件 → Promtail volume 挂载读取 → Loki。

宿主机命令审计：从 Bash History 到 eBPF

出事后想回溯"当时在机器上执行了什么"，需要命令审计。最简单的方式是采集 .bash_history——在 .bashrc 中加 PROMPT_COMMAND="history -a" 让每条命令立即 flush 到文件，Promtail 挂载文件采集。但它只能记录交互式 bash 命令，unset HISTFILE、换 shell、通过 Claude Code 等非交互式工具执行的命令都不会被记录，且每条命令触发一次磁盘写入。

更强的方案有两层：auditd（内核 2.6，在 execve 系统调用里埋硬编码钩子，无法绕过但不可编程、日志量大、不能跑在容器里）和 eBPF（内核 4.x+，动态加载 BPF 程序到内核 VM，可挂载任意内核函数，内核态过滤，容器化部署）。

最终选择 eBPF 方案（Tetragon，Cilium 出品）。

原理：execve 与 eBPF hook

Linux 上所有程序执行只有一条路：execve 系统调用。无论是 bash 敲命令、Claude Code 调 docker、cron 定时任务还是 Python subprocess.run()，最终都会调用 execve。但脚本内部的运算不会被记录——Python x = 1 + 1 不触发 execve，os.system("ls") 才会；Shell cd /tmp 不触发（bash 内置），ls 触发。

Tetragon 默认 hook 两个内核 tracepoint：sys_enter_execve（进程启动）和 sched_process_exit（进程退出）。

为什么能在 Docker 里运行

容器只是"发射台"——Tetragon 在容器内调用 bpf() 系统调用，把 BPF 字节码提交给宿主机内核，内核验证后加载到 tracepoint。之后 BPF 程序在内核态运行，跟容器无关。需要 privileged: true 是因为普通容器没有 bpf() 权限。删除容器时 BPF 程序随之卸载，宿主机零残留。

noDataState 的坑：两条规则的 noDataState 设置不同，原因很重要：

  • “Log pipeline service down” 用 noDataState: Alerting——没有 up 指标说明采集本身也坏了，应该告警。
  • “Promtail dropping entries” 用 noDataState: OK——没有 dropped 指标说明根本没在丢，不该告警。

容器生命周期告警

上面两条规则只覆盖了日志管道的健康，但更常见的场景是：某个容器挂了，没人知道。需要覆盖三种情况：容器宕机（停止/被杀）、容器异常重启、容器 healthcheck 变为 unhealthy。

为什么不用端点探测

业界最常见的做法是不监控容器状态本身，而是用 Uptime Kuma 或 HTTP probe 探测服务端点（比如 grafana:3000/api/health 是否返回 200）。配合 Docker 的 restart: unless-stopped，容器崩了自动重启，探针甚至感知不到。

但这个方案不适合我的场景：大量容器里很多服务没有 HTTP 端点（clash 代理、frpc 隧道、tetragon eBPF、redis、mysql 等），给每个服务配一个 TCP/HTTP 探针比容器层面的方案更麻烦。而且端点探测感知不到短暂重启和 OOM 等底层原因。

几个关键发现：

  1. cAdvisor 的盲区：cAdvisor 只上报运行中容器的指标，容器停了指标直接消失。无法用 PromQL 可靠表达"某个容器不在了"，因为 absent() 需要预知容器名。

排除临时容器: compose working_dir 过滤

常见的过滤思路（自定义 Docker label、restart_policy、compose_project）都有问题：要么需要侵入性地修改所有 compose 文件加标签，要么无法区分长驻服务和临时 benchmark（两者都有 restart policy 和 compose_project label）。

实际检查所有运行中容器的 com.docker.compose.project.working_dir label 后发现了一个天然的区分条件：

  • 临时 docker run --rm 容器：没有任何 compose label，天然排除
  • benchmark compose 容器：working_dir 为 /home/zombie/works/tsdb_bench，按路径排除
  • 长驻服务：所有其他 compose 容器

cAdvisor 配了 --store_container_labels=true，这些 Docker label 直接作为 metric label 出现在 VictoriaMetrics 里。PromQL 过滤条件：

零 label 侵入，零宿主机改动，新增长驻服务自动纳入监控。

最终方案: cAdvisor + docker events 混合

取长补短：cAdvisor 覆盖重启和 unhealthy（已部署，零新增），docker events 覆盖宕机（cAdvisor 的盲区）。

告警规则：

规则 数据源 表达式
容器异常重启 cAdvisor → VM changes(container_start_time_seconds{...working_dir!="", ...working_dir!~".*tsdb_bench.*"}[30s]) > 0
容器 unhealthy cAdvisor → VM container_health_state{...working_dir 同上} == 0
容器宕机 docker events → Loki sum by (actor_name) (count_over_time({job="docker-events",action="die"}[1m]))
容器 OOM 被杀 docker events → Loki sum by (actor_name) (count_over_time({job="docker-events",action="oom"}[1m]))

注意 docker events 的告警用 action="die" label 精确过滤，而不是 |= "die" 文本匹配。后者会误匹配 exec_die（exec 命令正常结束），导致大量误报。sum by (actor_name) 按容器名分组，告警消息里直接显示哪个容器出了问题。

单点静默：每条告警的 firing instance 都带 name label（容器名），在 Grafana Alerting 的 Silence 功能里按 name="xxx" 创建静默即可，不影响其他容器。

Telegram 通知

通知渠道：Telegram bot，group_wait: 10s、group_interval: 30s、repeat_interval: 4h。

group_wait 是首次告警等待聚合的时间，设短一些保证及时收到。group_interval 控制同组告警后续更新的间隔。这两个值直接决定了从容器挂掉到收到 Telegram 消息的延迟，实测从容器 die 到收到通知约 15~20 秒。

Grafana 容器通过 HTTP_PROXY 环境变量访问 Telegram API。注意代理的延迟需要低于 Grafana 通知的默认超时（5s），否则会 TLS handshake timeout。

应用层打点

上面讨论的都是被动采集——node_exporter 从内核接口读宿主机状态，cAdvisor 从 cgroup 读容器资源。这些都是系统视角，看不到业务视角：这个请求处理了多久？队列里积压了多少任务？某个 API 的错误率是多少？要回答这些问题，需要在代码里主动埋点。

总结与展望

当前成果

从零搭建了一套完整的可观测性栈，覆盖了 Metrics 和 Logs 两个信号：

  • 指标管道：node_exporter + cAdvisor → vmagent → VictoriaMetrics → Grafana，15 秒采集间隔，宿主机和全部容器的 CPU、内存、磁盘、网络指标全部可查可回溯。
  • 日志管道：Docker json-file → Promtail → Loki → Grafana，所有容器日志自动采集，按 compose_service 标签精确检索。额外采集 systemd journal、Tetragon eBPF execve 事件、Docker lifecycle 事件。
  • 告警：Grafana Unified Alerting + Telegram 通知，6 条规则覆盖日志管道健康、容器宕机、重启、unhealthy、OOM。告警规则全部通过 provisioning YAML 管理，不依赖 Grafana DB。
  • 安全：全容器 cap_drop: ALL + 最小权限 cap_add，no-new-privileges: true。

容器与服务清单

监控栈共 8 个容器 + 2 个宿主机 systemd 服务：

组件 类型 职责
node_exporter 容器 宿主机指标采集
cAdvisor 容器 容器指标采集、health_state
vmagent 容器 指标转发（scrape → remote_write）
VictoriaMetrics 容器 指标存储与查询
Promtail 容器 日志采集（Docker 日志、journal、tetragon、docker-events）
Loki 容器 日志存储与查询
Grafana 容器 可视化、告警引擎、Telegram 通知
Tetragon 容器 eBPF execve 审计
journal-export 宿主机 systemd 导出 systemd journal 为 JSON 文件
docker-events 宿主机 systemd 导出 Docker 生命周期事件为 JSON 文件

对宿主机的改动全部可通过 git 管理和追溯，全部放在 docker 配置的 git 仓库下，通过 symlink 链接到实际使用的位置。

后续计划

Promtail → Alloy 迁移：Promtail 已于 2025/02 宣布废弃，2026/03 EOL。继任者 Grafana Alloy（3100+ stars，v1.16.0，周更节奏）可以同时替代 Promtail 和 vmagent，单进程处理日志采集和指标转发。官方 Docker 镜像内置了 libsystemd，不再需要 journal-export 这种 workaround。缺点是内存占用较高（300Mi~1GiB vs Promtail 的 ~150Mi），有公开 issue 跟踪。当前不急于迁移，等内存问题改善后考虑，届时可以减少 2 个容器（Promtail + vmagent）和 1 个宿主机服务（journal-export）。

docker-state-exporter：当前容器宕机检测依赖 docker events + 宿主机 systemd 服务，是整个栈里唯一不能纯容器化的部分。docker_state_exporter 可以用纯容器方式暴露容器运行状态（包括已停止的容器），消除这个宿主机依赖。但项目成熟度一般（~150 stars，2023 年后低活跃）。本质上只是一个调用 Docker API 的小程序，后续可以考虑自己写一个，同时暴露 container state、health status、restart count，彻底去掉 docker-events systemd 服务。

应用层监控接入：当前监控覆盖了基础设施层（容器指标、日志、生命周期告警），下一步是接入应用层。计划中的几个方向：

  • Orth 任务调度监控：Orth（xxl-job 魔改版）是自研的分布式任务调度框架，用于交易数据的离线批量拉取。Grafana 通过 MySQL datasource 直接查询 orth_job_log 表，可以构建任务成功率、执行耗时、失败趋势等面板，和容器资源指标关联分析（比如任务失败时 CPU 是否打满）。Orth 自身的 admin UI 负责业务操作（触发、重试、日志查看），Grafana 负责运维视角的趋势监控和告警。
  • MySQL 监控：mysqld_exporter 读取 performance_schema、information_schema 和 SHOW GLOBAL STATUS，暴露连接数、QPS、慢查询、缓冲池命中率等指标。对 MySQL 零侵入，只需创建一个只读监控用户。
  • 交易数据存储监控：交易数据的存储和分析库正在选型评测中，确定后同样接入 Grafana 做查询性能和存储健康监控。

自定义业务面板：导入的社区仪表盘覆盖了通用指标，但看不到业务视角。后续计划搭建交易系统专属面板：K 线叠加买卖点、持仓盈亏曲线、策略信号触发频率。其他有趣的方向：qBittorrent 的下载速度和做种比、Clash 的流量分布、DST 服务器的在线玩家数。

关于 AI 接入：VictoriaMetrics 和 Loki 都提供 HTTP API，技术上完全可以让 AI agent 通过 PromQL/LogQL 查询指标和日志，辅助故障排查。已经尝试使用 SKILL 接入，但是目前尚未观测到实际收益，反倒是本次排查过程中，讲深入排查的思路做成了一个 NO-MAGIC-SKILL 非常好用。
