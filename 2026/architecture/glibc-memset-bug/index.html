|总字数:8k|阅读时长:31分钟|浏览量:

从一个 vector<vector<int>> 的写法习惯出发，深入 memset/memcpy 性能对比，用 perf 追进 glibc 汇编和源码，发现 AMD Zen 3/4/5 上 memset 的快速路径被意外封死，修复后性能提升 2 倍以上。9 轮调查，多次反转，最终提交了修复 patch。

本文由 AI 辅助写作，经作者审核修订后发布，文章 AI 含量：50%；代码 AI 含量 80%。

关于 No Magic Skill

本文的深挖过程由 no-magic-skill 思维框架驱动。这是一个为 Claude Code 设计的 skill plugin，核心理念来自南京大学蒋炎岩老师的教学哲学：计算机没有魔法，每一层都是人设计的，因此每一层都是可理解的。

它的起源是一次排查日志丢失的事件：8000 条 eBPF 事件到 Loki 后只剩 5 条，AI 连续给出三个看似合理但经不起验证的解释，直到强制要求对推理链中的每一个假设做实验验证，才找到真因（日志轮转）。由此提炼出五条公理：

  1. 计算机是可知模型 – 没有不可理解的黑盒
  2. 机器永远是对的 – 模型和行为不一致时，改模型不改机器
  3. 没有魔法 – 每个行为都有精确的技术原因
  4. 理论上可行就能做到 – 分解问题，搜索已有方案
  5. 永不满足 – 第一个解释很少是根因

本文没有严格遵循 no-magic 的状态机流程，但仅凭领会这个思想，AI 就在几乎无人监督的情况下完成了 9 轮深挖，从一个 3-5% 的性能差异一路追到 glibc 未公开的设计缺陷。no-magic 的核心不是流程，而是态度：数字对不上就继续挖，假设没验证就不算结论。

表面上都是"创建一个全零的二维数组"，但编译器看到的语义完全不同：

  • A：先构造一个 prototype vector<int>(COLS)，然后循环 N 次 复制 这个 prototype → 拷贝中执行 N 次 memcpy(dst, prototype, COLS*4)
  • B：循环 N 次，每行从空 vector 扩容 到目标大小，编译后每行执行一次 memset(dst, 0, COLS*4)

对全零 int 数组来说，memcpy 多做了一件完全无意义的事：从一个全零 buffer 读出零，再把零 写到目标。而 memset 只需要 写零。

Benchmark: 不同维度的完整对比

充分预热

多个 case 放在同一进程中顺序执行时，前一个 case 的 malloc/free 行为会改变 glibc 内部状态，污染后续 case 的测量结果。

原因是 glibc 的动态 mmap 阈值调整：malloc 对小分配用 brk()（arena 管理）对大分配用 mmap()，分界线默认 128KB。当 free 释放一个 mmap 块后，glibc 认为这个大小的分配可能会反复出现，将 mmap_threshold 提高到该块大小，后续同大小请求改走 brk() arena 以避免反复 mmap/munmap 的系统调用开销。这个自适应有上限：64 位系统上最高 32MB (DEFAULT_MMAP_THRESHOLD_MAX)。

Benchmark 一定要充分预热，但是 Claude 总是在这种地方踩坑，后面可以尝试找找什么 SKILL 去提升下意识。

反例深挖：为什么小行时 A 更快

语义复杂度分析

resize() 的语义是"扩容"，需要保留旧数据。即使 vector 是空的，编译器生成的代码仍然包含完整的扩容逻辑：

  1. 容量检查：现有容量是否足够？
  2. 增长因子计算：新容量 = max(旧容量 × 2, 目标大小)
  3. operator new 分配新 buffer
  4. memset 清零新元素
  5. 如果有旧数据，memcpy 搬迁 + free 旧 buffer
  6. 更新 begin/end/capacity 三个指针

这些逻辑可能被编译器内联，也可能作为独立函数 (_M_default_append) 调用，取决于编译器决策。但无论内联与否，步骤 1-2 的检查和步骤 5 的搬迁分支都会被保留：编译器无法在编译期证明 “vector 一定是空的”，所以无法消除这些冗余逻辑。

相比之下，A (fill constructor) 的语义是"用 prototype 填充 N 个元素"，没有"保留旧数据"的负担。每行的核心操作就是 operator new + memcpy（或 memmove）。同理，assign() 的语义是"全部替换"，emplace_back() 是从零构造新元素——两者都不需要保留旧数据，因此编译后不会产生搬迁分支。

A 和 B 的竞争本质是两个效应的交叉：

  • memset 优势：省掉读 prototype 的开销，和每行大小成正比（行越大优势越大）
  • resize 额外开销：容量检查、增长因子计算、旧数据搬迁分支等冗余逻辑，是近似固定的，和行大小无关

小行时固定项主导（A 快），大行时可变项主导（B 快），交叉点在 ~4KB 附近。

assign()、emplace_back()、operator= 的语义都比 resize() 简单：不需要保留旧数据，编译器能生成更精简的代码路径（直接 operator new + memset，无需容量检查和搬迁分支）。而一些 C++ 新特性带来的轻量视图，则是实现和原生二维数组几乎没有区别，是最推荐的高性能实现。

大 buffer 下 memcpy 为什么看起来比 memset 快

上面的只是一个写代码时候习惯的一个小问题探究，真正的问题在我们深入观察 memset 性能的时候才出现。

上面的隔离实验验证了 40KB 时 memset 比 memcpy 快 96%，符合"纯写比读+写快"的直觉。但把实验扩展到多个数据规模后（完整代码 test_memop.cpp），大 buffer 的结果看起来完全反转：

4 KB 1000000 27.894 ms 30.119 ms memcpy 慢 8%

数据呈现三个区间：

  1. 64B-256B（全在 L1 内）：持平。 L1 cache 的 load port 和 store port 是独立的，读写可以完全并行，memcpy 的额外读操作不构成瓶颈
  2. 1KB-4MB（超出 L1，在 L2/L3 内）：memcpy 慢 8%-133%。 数据需要经过 L2→L1 总线，读和写共享这条带宽，memcpy 用 2 倍带宽所以接近慢 2 倍
  3. 64MB-256MB（远超 L3）：memcpy 反而快 21-38%。 "纯写比读+写快"的直觉被推翻，后续分析会揭示原因

复现性确认：64MB 跑 3 次稳定在 memcpy 快 ~38%，256MB 稳定在快 ~21%。

测试环境：AMD Ryzen 7 8745HS (Zen 4)，L1d 32 KiB/core，L2 1 MiB/core，L3 16 MiB shared，Debian 13 trixie，glibc 2.41。

Round 1: perf stat 采集硬件计数器 | 微码循环和 SIMD

用独立测试程序 (test_memop_isolated.cpp) 隔离 memset/memcpy，便于 perf 分别采集：

64MB x 100 轮的基础指标：

指标 memset memcpy
时间 0.251s 0.158s
cycles 1,209M 744M
instructions 273M 857M
IPC 0.23 1.15

IPC (Instructions Per Cycle) 是 CPU 每个时钟周期 retire 的指令数，反映流水线利用率。现代 CPU 理论 IPC 上限 4-6，实际 1-2 算正常。0.23 意味着 CPU 每 4-5 个周期才 retire 1 条指令，绝大部分时间在空转。

memcpy 用了 3 倍的指令数，但 IPC 高出 5 倍，反而消耗更少的 cycle。memset 严重停滞。

初始假设：glibc 对 memset 和 memcpy 选择了不同的指令策略。x86 上实现 memset 有两条路径：

微码循环（rep stosb）：字面含义是"将 AL 寄存器的值重复写入 [RDI] 地址，重复 RCX 次"。早期 CPU 的微码实现真的逐字节操作，很慢。ERMS（Enhanced REP MOVSB/STOSB, Intel Haswell+ / AMD Zen+ 的 CPUID 特性）升级了微码：内部识别连续写入模式，以 cache line（64 字节）为单位批量处理，还能检测整条 line 被完全覆写时跳过 RFO（稍后介绍）、自动做 Write Combining（将多个小写合并为一次完整的 cache line 写入）。一条指令，零指令 fetch 开销，微码能做软件做不到的优化。对应的 memcpy 版本是 rep movsb。因为只有一条架构指令，perf 统计的 IPC 接近 0，但实际吞吐不一定差。

SIMD 显式循环：用 AVX-512 等向量指令（vmovdqa64），每条写 64 字节，需要软件写循环（load/store + add + cmp + jmp）。指令多，IPC 更容易衡量。

如果是微码循环，指令数应该极少（整个 64MB 只算几条），IPC 低是正常的。但这里有 273M 条指令，不符合微码的特征；如果是 SIMD 循环，273M 条指令说得通，但 IPC = 0.23 又异常地低。需要反汇编确认到底用了哪条路径。

Round 2: 反汇编定位热点指令 | Temporal 和 Non-Temporal

用 perf record 采样找到热点地址，再用 objdump 反汇编 glibc 对应位置：

memset 热点（采样 28% + 25% 集中在两条相邻指令）：

memcpy 热点（采样均匀分布在多条指令上）：

两边都是 AVX-512 SIMD 循环（每条指令操作 512 bit = 64 字节），Round 1 的微码假设被推翻：memset 用的不是 rep stosb，而是和 memcpy 类似的 SIMD 显式循环，但指令类型不同（temporal vs non-temporal）。指令数 273M（远多于微码循环的几条）也印证了这一点。

但两边都是 SIMD 循环，为什么 IPC 差 5 倍？关键差异在指令后缀——教科书上 cache 的写策略（write-back/write-through）是单核视角的简化模型，实际多核 CPU 上写 cache 之前必须先通过 MESI 协议拿到 cache line 的独占所有权，这个过程的代价远比"写一下 cache"大得多。因此现代 CPU 提供了两种 store 路径，由指令编码在发射时决定：

  • memset 用 vmovdqa64 (temporal store)：数据经过 cache 层级。CPU 和内存之间的数据传输以 cache line（x86-64 固定 64 字节）为最小单位。写一条不在 cache 里的 cache line 时，CPU 必须先发起 RFO (Read For Ownership)：从内存读回这 64 字节拿到独占所有权（多核一致性协议要求），然后才能写入新数据。对 memset 来说，读回来的旧数据会被立刻覆盖，这次读完全是浪费。写完后数据留在 cache 里，后续访问可以命中
  • memcpy 用 vmovntdq (non-temporal store)：数据绕过 cache，写入 WC buffer (Write Combining buffer)。NT 指令的本质是强制以 WC (Write-Combining) 内存类型协议处理这次写入，即使目标地址实际配置为 WB 类型。WC 协议的特性：不缓存、写合并、允许写乱序（因此 glibc 的 NT 循环结束后需要 SFENCE 保证可见性）。WC buffer 是 CPU 内部的小型硬件队列（Zen 4 上 4-8 个，每个 64 字节），多个写入同一 cache line 的 store 在这里合并，凑满一条完整的 64 字节 line 后一次性刷入内存，不需要先读回旧数据（没有 RFO）。代价是写完后数据不在 cache 里，后续读取要重新从内存加载

这里 vmovdqa64 每次写 64 字节恰好覆盖整条 cache line，但 MESI 协议仍要求先 RFO 拿到所有权才能写入。RFO 实际包含两步：(1) invalidate 其他核的副本（一致性要求，不能省）；(2) 从内存取回 64 字节旧数据（对 memset 来说是浪费）。能跳过第 (2) 步的只有两条路径：rep stosb ERMS 微码内部知道自己会顺序覆写整条 line，直接在 L1 分配空 line 写入，不取旧数据但数据仍留在 cache；NT store 则完全不走 cache 协议，写入 WC buffer 后刷到内存。普通单条 store 指令做不到这个优化，因为硬件发射时无法确定整条 line 的所有字节都会被覆写。

64MB 远超 L3 (16MB)，每条 cache line 都不在 cache 里。memset 的每个 store 都触发一次无意义的 RFO 内存读。RFO 延迟 ~40-100ns，等待期间 store 堆积在 store buffer（CPU 内部的 pending store 队列，Zen 4 上 64 条目）里。store buffer 满了之后，后续所有 store 指令都无法发射，流水线停滞。这就是 IPC = 0.23 的直接原因。

memcpy 的 non-temporal store 不需要等任何东西，直接进 WC (Write Combining) buffer 然后写入内存。流水线不堵，IPC 保持 1.15。

Round 3: 验证 RFO 假设 | Temporal 指令导致 RFO 拖慢性能

用 L1-dcache-load 计数器验证 memset 是否真的在做大量隐式读操作：

指标 memset memcpy
L1-dcache-loads 246M 465M
L1-dcache-load-misses 106M (43%) 80M (17%)
L1-dcache-prefetches 51M 0.2M

memset 是纯写操作，却产生 246M 次 L1 load 和 106M 次 load miss。这些 load 不是程序指令发起的，而是 CPU 微架构为 temporal store 自动插入的 RFO 请求。RFO 假设成立。

Round 4: 为什么 memset 不用 non-temporal store

既然 non-temporal 明显更快，glibc 为什么不用？反汇编发现 memset 内部有一个阈值判断，超过阈值才跳转到另一条路径。用 GDB 读取运行时阈值：

阈值是 0xFFFFFFFFFFFFFFFF (SIZE_MAX)。无论 buffer 多大，条件 size >= SIZE_MAX 永远为 false。non-temporal 路径被完全禁用。

而 memcpy 没有这个问题，始终对大 buffer 使用 vmovntdq。问题出在 memset 的阈值配置上。

Round 5: glibc 源码追踪 | 嵌套条件的意外交互

glibc 的 memset 根据 buffer 大小选择三条路径，由两个独立阈值控制：

  • rep_stosb_threshold：用微码循环还是 SIMD 循环（两者在 Round 1 已介绍）。Intel 的 ERMS 微码优化激进，中等 buffer 就开始赢 SIMD; AMD Zen 3+ 的 SIMD 向量单元更强，SIMD 在所有大小上都赢
  • non_temporal_threshold：要不要绕过 cache。小于 L3 时数据应留在 cache（后续访问快）; 大于 L3 时绕过 cache 省掉 RFO

正常情况下 stosb_thresh < nt_thresh，运行时判断结构：

四层赋值链追踪到问题：

第一层：AMD 正确清除了 Avoid_Non_Temporal_Memset，内层判断 non_temporal_threshold 被赋予真实值 (~12MB)。

第二层：AMD Zen 3+ 的 SIMD 循环比 rep stosb 快，glibc 将 rep_stosb_threshold = SIZE_MAX 禁用 stosb。SIZE_MAX 让外层 if 永远不成立，嵌套在内部的 non-temporal 路径被一起封死。

第三层：glibc 有补救代码，但条件是 Avoid_STOSB（CPU 不支持 ERMS 硬件）。AMD Zen 4 硬件上支持 ERMS，只是软件选择不用，所以补救不执行。

Round 6: 多层级硬件计数器验证

用 AMD PMU 从 L3、UMC（内存控制器）、L1 fill 三个层级全面采集：

层级 指标 memset memcpy 比值
L3 accesses 129.3M 12.4M 10x
miss (→DRAM) 112.5M (87%) 8.0M (64%) 14x
UMC CAS read (DRAM→CPU) 53.8M 2.8M 19x
CAS write (CPU→DRAM) 49.4M 53.5M 0.92x
L1 fill 总 fill 106.5M 1.9M 56x

memset 的 106.5M 次 L1 fill 和预期吻合 (64MB/64B x 100 = 100M cache lines)，每条 line 走完整的 DRAM→L3→L2→L1 路径 (RFO)。

memcpy 在所有层级都异常低：L1 fill 只有 1.9M，预期应该和 memset 一样是 ~100M。结合 Round 3 数据（memcpy L1 命中率 83%），实际原因是 prefetcht0 提前将 src 数据填充到 L1，后续 vmovdqu64 load 直接命中 L1 而不触发额外的 demand fill。L1 fill 计数器低是因为 prefetch 引起的填充不被该 PMU 事件计数，不是数据绕过了 cache。

实际 DRAM 总流量两边接近（memset 因 RFO 读 ~6.4GB + 写回 ~6.4GB; memcpy 读 src ~6.4GB + NT 写 dst ~6.4GB），但 memset 慢 60%。瓶颈不在总带宽，而在 RFO 阻塞：temporal store 必须等 cache line ownership 才能 retire，store buffer 堵满后流水线停滞; memcpy 的 prefetch 提前拉数据、NT store 直接写出，全程不阻塞。

Round 8: non-temporal 并不总是更快 | Page Fault 的副作用

用 workaround 重新跑 vector 初始化 benchmark，发现一个意外：

场景 temporal (默认) non-temporal (修复) 变化
有 page fault (首次分配) 109.7 ms 127.0 ms NT 慢 16%
无 page fault (预热后) 14.2 ms 5.6 ms NT 快 2.5x

有 page fault 时 NT 反而更慢？用 perf stat 分离 user/sys 时间：

指标 temporal non-temporal
user time 0.008s 0.032s
sys time 0.134s 0.127s

temporal 路径用户态只花了 8ms，几乎免费。原因是 page fault 的处理流程：

  1. 用户态 memset 写到未映射的虚拟页 → CPU 触发 #PF 异常，陷入内核
  2. 内核分配物理页，清零（安全要求：防止进程读到其他进程数据）
  3. 内核调 clear_page_erms (rep stosb) 清零 4KB。为什么不用 SIMD？内核里用 AVX-512 需要先保存用户态的 ZMM 寄存器 (32 x 64B = 2KB)，保存/恢复开销远超 SIMD 对 4KB 的收益。rep stosb 只用 RAX/RCX/RDI 三个通用寄存器，零额外开销。64 条 cache line 以 Modified 状态进入 L1
  4. 返回用户态，memset 继续写这一页 → 全部命中 L1

page fault 逐页触发，每页的 clear_page 和用户 memset 背靠背发生，cache line 还没来得及被驱逐就被覆写了。temporal memset 免费搭了内核 clear_page 的便车; non-temporal 绕过了这趟便车，强行走内存路径。

但这趟便车只在首次分配时存在。预热后的后续轮次没有 page fault，cache line 全部冷掉，temporal 的 RFO 打穿到 DRAM，反而比 NT 慢 2.5 倍。

Round 9: 最终对比

回到最初的问题：前面多规模实验中 64MB/256MB 时 memcpy 比 memset 快，是真的吗？那组实验中 memset 和 memcpy 在独立进程中运行，cache 状态不同：memcpy 的 NT write 不污染 cache，src 数据能驻留 L3 被后续迭代复用，相当于额外占了便宜。在同一进程中控制相同 cache 状态后 (test_final_fair.cpp):

操作 预 fault (64MB x 100) 有 PF (64MB x 100)
默认 glibc
memset(0) temporal 2.297 ms 18.229 ms
memcpy non-temporal 4.399 ms 23.118 ms
修复后
memset(0) non-temporal 0.881 ms 20.988 ms
memcpy non-temporal 4.253 ms 22.931 ms

控制相同 cache 状态后，memset 始终比 memcpy 快。 "纯写比读+写快"的基本原理成立。修复后预 fault 场景 memset 最快 (0.881ms)，是 memcpy 的 4.8 倍。

缺陷溯源

这个缺陷是两个独立 patch 的意外交互：

  1. 2024-02, BZ 30994：修复 Zen 3/4 上 ERMS 比向量循环慢的问题，设 rep_stosb_threshold = SIZE_MAX
  2. 2024 Q3, non-temporal memset for AMD：为 AMD 启用 non-temporal memset（清除 Avoid_Non_Temporal_Memset）

修复 1 禁用了 stosb（因为 AMD Zen 上 AVX-512 SIMD 循环吞吐始终优于 rep stosb 微码），但 non-temporal 路径嵌套在 stosb 分支内部，被一起封死。修复 2 正确启用了 non-temporal 阈值，但没注意到入口已经被修复 1 封死。两边各自测试都通过了，合在一起才出问题。影响范围：所有支持 ERMS 的 AMD Zen 3/4/5 CPU，glibc >= 2.39。

修复方案

我其实更倾向于直接把嵌套 IF 摊平，但是 Claude 建议第一次就提交一个最小的 Patch，避免造成更大的影响。sysdeps/x86/dl-cacheinfo.h:1296，扩展现有的补救条件：

三个条件：原条件 Avoid_STOSB（硬件不支持 ERMS）; 新增 !TUNABLE_IS_INITIALIZED（stosb 阈值不是用户手动设的，尊重显式配置）; 新增 > 比较（stosb 门控确实挡住了 NT 入口）。当 NT 被禁用时 memset_non_temporal_threshold == SIZE_MAX，rep_stosb_threshold > SIZE_MAX 对 unsigned long 恒为 false，条件自然不触发，不需要额外检查。

提交前我反复推敲过 TUNABLE_IS_INITIALIZED 这个条件。第一反应是去掉它: rep_stosb_threshold 控制的是 stosb 行为，不该连带卡 NT 路径，用户设 stosb 阈值并不意味着想禁用 NT。但仔细看上游 BZ 30994 的代码，rep_stosb_threshold = SIZE_MAX 赋值本身也有同样的保护:

上下两处 TUNABLE_IS_INITIALIZED 形成对称逻辑: 用户没设 tunable 时，上面赋 SIZE_MAX，下面降回 NT 阈值，fix 生效; 用户显式设了 tunable 时，上面不覆盖用户值，下面也不介入，用户的设置原样保留。如果用户设了一个小值（比如 1MB < NT 阈值 ~12MB），stosb 外层 if 本来就能进入，NT 路径自然可达，不需要 fix; 如果用户显式设了 SIZE_MAX，那是用户知道自己在做什么。不同平台上 rep_stosb_threshold 对 NT 路径的门控语义也保持一致: 用户显式设高阈值 → stosb 和 NT 都不走，跟非 AMD 平台行为相同。

总结

从一个 LeetCode 写法习惯出发，经过 9 轮深挖，最终产出了一个 glibc patch (BZ #34129)。中间涉及 Benchmark、汇编、源码分析、内核、glibc 等各个层次，最终定位到 glibc 的一个设计缺陷：由于嵌套 IF 条件导致 AMD 处理器上高性能 NT 语句不可达。
我本更倾向于直接彻底修复嵌套 IF，但是 claude 担心影响更大，于是最小改动 patch 让 NT 路径可达，根本的彻底解耦留给维护者去评估和完成。

发现这个问题的时候非常兴奋，当时反复确认是否是一个真实存在的 bug，按程序员的直觉：使用的人越多出错概率越小，我总是错的。但是最终验证确实是一个性能 bug，并且在 glibc 存在 2 年之久。
这个优化虽然能将性能提升到两倍以上，但在 page fault 的场景反而略慢。而多数应用的 memset 调用远远达不到 64 MB 的大小，实际收益可能非常小。

最终成功被合入：https://gitlab.com/gnutools/glibc/-/commit/54abc8566fea592e795cb443949266ef206462a8

glibc 的 x86 string/memory 优化代码主要由 Noah Goldstein（Citadel Securities，业余时间贡献）和 H.J. Lu（前 Intel x86 后端负责人，已退休后活跃度降低）编写，review 和 commit 则由更广泛的维护者团队负责，包括 Adhemerval Zanella（Linaro，全职维护 glibc）等。这和很多关键基础设施项目的处境类似：全世界都在用，靠少数人扛着。

后话：共享 0 页的助攻

在 Claude Max 于 2026 年 6 月 30 日被封禁之后，我改用 Claude Code + GLM 5.2。在测试模型能力的时候，让 AI 独立复现"memcpy 比 memset 快"这个异常时，它一开始得出了相反的结论：绝大多数情况下 memset 仍然比 memcpy 快。

简单深挖了下，发现还有一个前文没有提到的原因：编译器把 malloc + memset(src, 0) 折叠成了 calloc，src 一直没有被 touch，落在 内核共享零页上，memcpy 的读侧全部命中 L1，才让 memcpy 反超。把填充的内容换成其他数值，memcpy 的速度还是比有 bug 的 memset 更慢。
