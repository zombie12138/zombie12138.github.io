跟踪动态链接库 Trace 的各种思路

发表于2024-03-18|更新于2026-04-28|Kernel&OS
|总字数:6.2k|阅读时长:25分钟|浏览量:

最近想要 Trace 一下TVM生成库的调用, 于是使用 ltrace, 但是发现 ltrace 不会去追踪 dlopen + dlsym 得到的函数地址.

1 问题

最近在写一下这个 TVM 的 CPP 的手动推理, 不依赖于 TVM 自身 Runtime 的. 想要验证正确性, 除了推理的结果, 还想要看一下 trace 的不同, 于是使用 ltrace, 但是由于库是在运行时通过 dlopen 直接加载的, 所以无法直接 Trace 到.

1.1 编译时的动态链接 | GOT | PLT

编译器生成的会在代码的符号中有所体现, 由于 ASLR (Address space layout randomization)[1] 这种安全机制, 我们无法确定动态链接库装载的地址. 我们需要一些机制来在运行时确定变量和函数的地址. PLT (Procedure Linkage Table) 和 GOT (Global Offset Table) 是一起协同发挥作用的. 程序启动后, 动态链接器修改 GOT 和 PLT. **GOT 用来记录符号对应载入后的地址, 而PLT则是使用GOT中存储的偏移量来决定函数的实际最终位置**[2]. 也太抽象了, 我们来看一个例子:

plt, 会通过 got.plt 执行**跳转**到真正的代码, 在第一次执行的时候会触发动态链接器去解析地址, 之后的调用可以直接 jmp. ltrace 的原理就是通过 hook GOT 表来实现函数调用追踪. 发现 gdb, ltrace 和 strace 都会影响 ltrace 的执行, 出现 `PTRACE_TRACEME: Operation not permitted`, 这个似乎是 ptrace 本身权限的问题. 此外, 写了一个获取自身的 `dl_iterate_phdr` 遍历的程序, 在使用 ltrace 的时候也是会出问题, 应该是 `dl_iterate_phdr` 和 `ltrace本身发生了冲突`. ### *1.2* dlopen 但对于通过 `dlopen` 加载后再使用 `dlsym` 获取函数地址并直接调用的情况, ltrace 则无法追踪. ## *2* Trace的思路 从复杂到简单. ### *2.1* 内存权限和 SIGSEGV 这个是从 PPoPP14 的一篇论文里面 HOOK NUMA 内存访问看到的一个思路[6]. - 加载库之后 `mprotect` 去修改页面权限为不可执行 - 之后 `sigaction` 之类的注册一个处理函数 - 捕获到 SIGSEGV 之后, SIGNAL 处理函数 1. 设置页面权限为可执行 2. 记录 Trace 3. 执行函数 4. 记录 Trace 5. 设置页面权限为不可执行 6. 返回, 回到正常流程运行 其他的都可以很好的执行, 但是问题在于最后步骤 `6`, 如果返回的话还是返回到函数调用的时候, 所以如果上面的流程还是会触发之前的中断, 也就是死循环了. 可以试试例子. 但是如果在步骤 `2` 就恢复页面权限就返回到正常流程, 那么这时候页面权限又开放了, 无法拦截调用. 想看一下 PPoPP 论文里面怎么做的, 但是他的内存访问是直接插桩做的, 只有 FirstTouch 识别才是用这个内存权限来做的, 来捕获 page fault. 但是显然不适用于我们这里. ### *2.2* 编译器插桩 就是在函数进入和离开的时候手动插入一些 "桩", 这里用来记录 trace. #### *2.2.1* finstrument-functions 那直接编译器插桩做的呢, 这个其实早就有了 之前的例子: 使用 `__cyg_profile_func_enter` 和 `__cyg_profile_func_exit`, 同时编译时候使用 `-finstrument-functions` 就可以了, 但是可以看到结果只有main函数中插入了这些指令, 如果要在 add 中使用, 必须编译时候在 `add.c` 之前就定义这两个函数, 并且在编译 `libadd.so` 的时候使用 `-finstrument-functions` 编译选项. 其原理也非常简单, 就是编译时候在自己的代码里面开头和结尾直接插入这两个函数的调用. 所以必须 **修改库文件和编译指令**. #### *2.2.2* LLVM: 直接写一个 PASS 上面这个需要修改源代码, TVM 虽然也可以修改一下代码啦, 但是还是不方便的, 最好的可能就是直接 llvm 本身注册加入一个 pass, 这样甚至不需要修改编译选项[7]. 然后使用下面的 Makefile 构建完成之后, 对于一个文件: 这个可能自由度大一些, 但是对于 TVM 就不方便了, 虽然输出可以给出来 llvm, 但是需要手动执行 pass, 手动编译这个 ir. 先生成模型和模型代码的 ir: 之后再执行下面的操作, 使用 llvm 的 pass 去编译 ir, 然后放到打包里面: 好啦, 我们用之前 TVM 编译流程里生成的输入文件来运行： 功能实现了, 但是有很多 lambda 函数都被记录在里面了, 如果真的要用可能还是需要去过滤输出或者在 pass 中直接过滤. TVM 的 IR 不知道能否支持这种调用, 但是其 API 本身就一直变动, 就没做了. ### *2.3* 动态二进制插桩 和 静态代码重写 上面的需要源码, 需要修改编译选项, 如果使用了一些编译框架, 比如 TVM, 还需要手动修改其中的代码甚至框架代码. 二进制最大优点就是不需要源码, 不依赖使用的技术路线, 但是会依赖目标平台和架构. 有一些工具, 比如 Intel 的 Pin[9], Frida, 但是他们似乎是运行时的插桩. 我想要找静态的插桩, 或者说静态代码重写, 主要是感觉好用一点, 似乎有 Radare2, Dyninst 什么的, IDA 这些似乎也可以, 但是感觉可能会比较复杂. 目前不清楚有什么好的的静态的二进制重写的工具, 手动更复杂: 1. 根据 elf 文件格式找到函数名 2. 插入指令, 压入一些自己要用的寄存器, 来保存现场 3. 插入调用 printf 的指令, 还是按照动态库的形式. 4. 恢复现场 5. 一些对齐和段表之类的修改 6. 可能要修改字符串表, 比如要打印的字符串, 以及一些其他的东西. 因此暂时不实现. ### *2.4* hook dlsym 这个是最方便的, 这个不需要源码, 不算二进制的注入, 因为没有修改二进制的任何字节, 而是直接hook运行时, 也算是动态的. 思路大概是: 1. 加载 ld 的 `dlsym` 的地址 2. 编写一个 proxy 或者 dummy 的 `dlsym` 的函数去 hook 这个过程, 再这个函数中: 1. 每次调用都会检查有没有一个可写的空间, 如果没有, 就创建一个新的, 并且设置 rwx, 这个空间可以组织成链表, 每个节点是一个页面, 页面用完就返回. 2. 调用实际的 `dlsym`, 获得真正的地址 3. 生成代码, 写入内存, 这个代码会调用 printf 函数名 (函数名也存在一个单独的字符串表里面), 之后跳转到实际的地址 3. 把这个 `dlsym` 变成一个动态链接库 `libhookdl.so`, 去抢先注册 dlsym 这个名称. 这样, 每次调用 `dlsym`, 如果是目标的, 感兴趣的库, 都会生成一个包装真实调用的函数, 返回其地址. 而每一次调用这个函数都是调用被 hook 的, 被包装的函数. 使用 `LD_PRELOAD` 来抢先这个符号的解析, 是很常见的方法, 但是问题在于我们的这个 `libhookdl.so`, 也是需要 `dlsym` 的. 如果下面这种做法就会直接 StackOverflow, 拿不到 libdl.so 的dlsym 我想的解决方法是, 二进制重写 libdl.so 的 `dlsym` 名字为 `dlsym2`, 然后新的库叫 `libld2.so` (同名也没有关系, 第二次是否加载 `libld.so`也不会影响符号的解析), 让这个 `libhookdl.so` 使用这个里面的 `dlsym2`, 并且注册 `dlsym`. 但是 GPT 给出了更好的解决的办法: > 1. 使用静态链接: 在你的hook库中直接静态链接 dlsym 的实现. 这种方法可能需要你访问到libc的源代码或者一个能够提供 dlsym 功能的静态库. 这种方法相对复杂, 不推荐. (这个方法不太能实现) > 2. 使用 __libc_dlsym, 有些系统的libc提供了 __libc_dlsym 函数, 这个函数和 dlsym 功能相同, 但是可以用来避免和 dlsym 的直接依赖. 这是一个内部函数,并不在所有系统上都可用, 使用时需要谨慎. > 3. 利用已知的偏移量: 如果你知道 dlsym 函数在其所在共享库 (通常是libc) 中的偏移量, 你可以计算出其地址. 这需要你首先找到共享库的基地址. 这种方法和具体的系统实现紧密相关, 具有一定的风险性和不可移植性. > 4. 最常用的解决方案 — 间接引用: 一种更常用和相对安全的方法是在你的 hook 库中定义一个指向 dlsym 函数的指针，并在库被加载时，通过其他手段 (比如在一个构造函数中使用 /proc/self/maps 来扫描内存映射并找到libc及其符号) 来解析和设置这个指针的值. 一旦这个指针被正确设置, 你就可以通过它来调用原始的 dlsym 函数了. 所以直接使用了libc的 `__libc_dlsym` 去代替 `dl.so` 的 `dlsym`. 下面就是 hook 的代码 编译成 bin, 拿到 cpp 里面, 每次 cpp 都插入上面 asm 编译出来的二进制 但是...这个可以成功hook一些简单的例子, tvm 的 cpp 的推理会提示 bad allocation, python 的一些库 import 执行 init 的时候会报错. 这个存在一些已知的问题 - 过滤变量: 除了过滤 lib 库名, 还有就是动态库的变量, 也需要检查是变量 or 函数, 如果是变量就不要 hook. - 上下文保存: 我们这里只保存了几个用到的寄存器, 但是有 **很多调用参数的寄存器也需要保存**, 比如 `xmm` 这些, 调用者保存寄存器都需要保存. 但是尝试保存了更多的寄存器之后, 哪怕使用 malloc 这样的有时候也会出 bug, 这个 debug 太痛苦了. - `r15`保存逻辑: 这里面的保存 `r15` 的过程也会存在问题，因为如果调用之后又再次调用, 这时候就会覆盖这个 r15 的值, 解决办法也很简单, 设置一个表就可以了, 但是 `r15` 不是一个调用者保存的寄存器, 猜测就算正确保存了之后依然可能出问题, 就没有改 bug 上面的过程不是最佳实践, 肯定会有更方便的二进制 hook 和生成二进制代码的工具. ### *2.5* TVM API 一般来说这些框架都会提供 profiler 或者 tracer 之类的功能. 但是搜索了一下 trace, debug, profile, 这些关键词大概找了一下只有 [PAPI](https://tvm.apache.org/docs/how_to/profile/papi.html) 来 Profile 算子, 但是似乎没有 trace. ### *2.6* gdb 等工具 gpt 给我写的是自动 nm, 然后自动设置断点, 这种自动化思路虽然可行, 但效率太低. 又建议使用 bbc, 这个不是很熟悉, gpt 又写了一个牛头不对马嘴的脚本, 也比较复杂, 暂时不是嫌 ## *3* 总结 想要追踪 TVM 生成的库的调用的 Trace, 由于 tvm 是使用 `dlopen` 这种方法, 但是 ltrace 不起作用. 我们通过以下几个思路 1. 设置内存权限, 然后捕获 SIGNAL: **并不可行**, 只可以捕获这个页面第一次的调用 2. 编译器插桩: 可以通过编译器自带的或者 IR, tau (用来性能分析) 就是这样做的, 但是这个需要源码, 这个是 **最后唯一能用的** 3. 二进制重写: 不需要源码, 但是需要依赖目标平台, 而且做不好会出很多运行时的问题, **太复杂** 4. Hook dlsym: 每次载入符号, 载入自己的包装之后的函数, 也是相当于运行时的注入插桩, **还蛮有意思**, 但是我手动实现的这一版本 bug 太多, 调试困难. 5. TVM trace: 按说应该有现成的, 但是没仔细找, 但是 **没找到**. 6. gdb 等其他工具 ## 参考文献
