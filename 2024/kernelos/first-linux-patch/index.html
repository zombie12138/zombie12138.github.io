总结

同样的错误 drivers 下面还有很多, 感觉可能会干扰 reviewer 的正常工作, 暂时不发了 hhhh. 作为一个没有尝试过的还是值得一试的, 毕竟可能很多人也是 TYPO 做起.

感觉就是浪费人家时间. 这群人效率真的很高. 而且很耐心

  • Linux commit 规约一般是: subsystem: imperative sentence, 提交之前的 subsystem 需要查名字可以直接看一下 commit 日志, 看一下别人是怎么写的.
  • 如果需要重新提交补丁, 不应 reply 之前的邮件, 而是应该发送一封全新的邮件.
  • 所有的工作流都有对应的脚本, 比如找收件人, 完全可以自动进行.

这样看来, 国内某公司通过一些自动检查 typo 的工具, 自动生成 patch 刷 KPI, 没有很复杂.
