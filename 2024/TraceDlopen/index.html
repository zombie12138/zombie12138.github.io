跟踪动态链接库Trace的各种思路

发表于 2024-03-18 更新于 2024-04-09

最近想要Trace一下TVM生成库的调用, 于是使用ltrace, 但是发现ltrace不会去追踪dlopen打开, 之后dlsym得到的函数地址.

问题

最近在写一下这个TVM的CPP的手动推理, 不依赖于TVM自身Runtime的. 想要验证正确性, 除了推理的结果, 还想要看一下trace的不同, 于是使用ltrace, 但是由于库是在运行时通过dlopen直接加载的, 所以无法直接Trace到.

编译时的动态链接 | GOT | PLT

编译器生成的会在代码的符号中有所体现, 由于ASLR (Address space layout randomization)[1]这种安全机制, 我们无法确定动态链接库装载的地址. 我们需要一些机制.

PLT(Procedure Linkage Table)和GOT(Global Offset Table)是一起协同发挥作用的. 程序启动后, 动态链接器修改GOT和PLT. GOT用来记录符号对应载入后的地址, 而PLT则是使用GOT中存储的偏移量来决定函数的实际最终位置.[2]

总结

虽然没仔细看ld.so的代码, 但是可以知道, 动态链接库存在GOT, 这里存放着全局变量和函数的真正的地址; plt, 会通过got.plt执行跳转到真正的代码, 在第一次执行的时候会触发动态链接器去解析地址, 之后的调用可以直接jmp.

ltrace就是去hook这个got

发现gdb, ltrace和strace都会影响ltrace的执行, 出现PTRACE_TRACEME: Operation not permitted, 这个似乎是ptrace本身权限的问题.
此外, 写了一个获取自身的dl_iterate_phdr遍历的程序, 在使用ltrace的时候也是会出问题, 应该是dl_iterate_phdr和ltrace本身发生了冲突.

dlopen

我们知道了ltrace是通过got这部分来处理, 但是对于dlopen的库再去dlsym直接拿到地址存到一个指针里面, 就没法追踪, 尽管wiki说0.73版本可以, 但是实测是不行的, 似乎也很搜不到太多的回答.

几个思路

从复杂到简单.

内存权限和SIGSEGV

这个是从PPoPP14的一篇论文里面HOOK NUMA内存访问看到的一个思路[6].

  • 加载库之后mprotect去修改页面权限为不可执行
  • 之后sigaction之类的注册一个处理函数
  • 捕获到SIGSEGV之后, SIGNAL处理函数
    1. 设置页面权限为可执行
    2. 记录Trace
    3. 执行函数
    4. 记录Trace
    5. 设置页面权限为不可执行
    6. 返回, 回到正常流程运行

其他的都可以很好的执行, 但是问题在于最后, 如果返回的话还是返回到函数调用的时候, 所以如果上面的流程还是会触发之前的中断, 死循环. 可以试试例子.

但是如果在步骤2就恢复页面权限就返回到正常流程, 那么这时候页面权限又开放了, 无法拦截调用.

可以的一个想法就是通过特定的编译/链接手段? 每一段最后都添加一个跳转, 跳转到删除so库内存执行权限的代码, 把之前的代码重新封印之后再返回. 如果这样的话, 就可以直接去得到Trace了, 因为我们可以直接Hook每一次调用, 不需要这么麻烦了… 后面我们也这样做了.

想看一下PPoPP论文里面怎么做的, 但是他的内存访问是直接插桩做的, 只有FirstTouch识别才是用这个内存权限来做的.

动态二进制插桩和静态代码重写

上面的需要源码, 需要修改编译选项, 如果使用了一些编译框架, 比如TVM, 还需要手动修改其中的代码甚至框架代码.

二进制最大优点就是不需要源码, 不依赖使用的技术路线, 但是会依赖目标平台和架构. 有一些工具, 比如Intel的Pin[9], Frida, 但是他们似乎是运行时的插桩.

我想要找静态的插桩, 或者说静态代码重写, 主要是感觉好用一点, 似乎有Radare2, Dyninst什么的, IDA这些似乎也可以, 但是感觉可能会比较麻烦…
似乎没有特别方便好用的静态的二进制重写的工具, 手动更复杂

  1. 根据elf文件格式找到函数名
  2. 插入指令, 压入一些自己要用的寄存器, 来保存现场
  3. 插入调用printf的指令, 还是按照动态库的形式.
  4. 恢复现场
  5. 一些对齐和段表之类的修改
  6. 可能要修改字符串表, 比如要打印的字符串, 以及一些其他的东西.

这个后面再做吧

hook dlsym

这个是最方便的, 这个不需要源码, 不算二进制的注入, 因为没有修改二进制的任何字节, 而是直接hook运行时, 也算是动态的. 思路大概是:

  1. 加载ld的dlsym的地址
  2. 编写一个proxy或者dummy的dlsym的函数去hook这个过程, 再这个函数中:
  3. 每次调用都会检查有没有空间, 如果没有, 就创建一个新的, 并且设置rwx, 这个空间可以组织成链表, 每个节点是一个页面, 页面用完就返回.
  4. 调用实际的dlsym, 获得真正的地址
  5. 生成代码, 写入内存, 这个代码会调用printf函数名(函数名也存在一个单独的字符串表里面), 之后跳转到实际的地址
  6. 把这个dlsym变成一个动态链接库libhookdl.so, 去抢先注册dlsym这个名称.

这样, 每次调用dlsym, 如果是目标的, 感兴趣的库, 都会生成一个包装真实调用的函数, 返回其地址. 而每一次调用这个函数都是调用被hook的, 被包装的函数.

问题在于, 使用LD_PRELOAD来抢先这个符号的解析, 是很常见的方法,
但是问题在于我们的这个libhookdl.so, 也是需要dlsym的…
这个hook的函数就是dlsym本身, 没法使用dlsym去hook…

我想的解决方法是, 二进制重写ld.so的dlsym名字为dlsym2, 然后新的库叫libld2.so(同名也没有关系, 第二次是否加载libld.so也不会影响符号的解析), 让这个libhookdl.so使用这个里面的dlsym2, 并且注册dlsym. 但是GPT给出了更好的解决的办法:

  1. 使用静态链接：在你的hook库中直接静态链接 dlsym 的实现。这种方法可能需要你访问到libc的源代码或者一个能够提供 dlsym 功能的静态库。这种方法相对复杂，不推荐。
  2. 使用 __libc_dlsym：有些系统的libc提供了 __libc_dlsym 函数，这个函数和 dlsym 功能相同，但是可以用来避免和 dlsym 的直接依赖。这是一个内部函数，并不在所有系统上都可用，使用时需要谨慎。
  3. 利用已知的偏移量：如果你知道 dlsym 函数在其所在共享库（通常是libc）中的偏移量，你可以计算出其地址。这需要你首先找到共享库的基地址。这种方法和具体的系统实现紧密相关，具有一定的风险性和不可移植性。
  4. 最常用的解决方案 — 间接引用：一种更常用和相对安全的方法是在你的hook库中定义一个指向 dlsym 函数的指针，并在库被加载时，通过其他手段（比如在一个构造函数中使用 /proc/self/maps 来扫描内存映射并找到libc及其符号）来解析和设置这个指针的值。一旦这个指针被正确设置，你就可以通过它来调用原始的 dlsym 函数了。

TVM API

一般来说这些框架都会提供profile或者trace之类的功能. 但是搜索了一下trace, debug, profile, 这些关键词大概找了一下只有PAPI来Profile算子, 但是似乎没有trace.

gdb等工具

gpt给我写的是自动nm, 然后自动设置断点, 这也太笨了. 又建议使用bbc, 这个不是很熟悉, gpt又写了一个牛头不对马嘴的脚本, 还是先算了吧

总结

想要追踪TVM生成的库的调用的Trace, 由于tvm是使用dlopen这种方法, 但是ltrace不起作用(虽然Wiki上说可以). 我们通过以下几个思路

  1. 设置内存权限, 然后捕获SIGNAL: 并不可行, 只可以捕获这个页面第一次的调用
  2. 编译器插桩: 可以通过编译器自带的或者IR, tau(用来性能分析)就是这样做的, 但是这个需要源码, 这个是唯一一个最后能用的
  3. 二进制重写: 不需要源码, 但是需要依赖目标平台, 而且做不好会出很多运行时的问题, 懒得做
  4. Hook dlsym: 每次载入符号, 载入自己的包装之后的函数, 也是相当于运行时的注入插桩, 还蛮有意思, 就是不好用.
  5. TVM trace: 按说应该有现成的, 但是没仔细找, 就没找到… 而且这个API更新太快.
  6. gdb等其他工具

每次都跑偏, 但是还很喜欢跑偏的感觉, 不过真的要改改了…
