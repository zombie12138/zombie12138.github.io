udp2raw 的使用 | 增加DNS功能

发表于2024-05-02|更新于2025-12-23|Networking
|总字数:2.6k|阅读时长:11分钟|浏览量:

最近使用 Wireguard 来利用公网跑 qbittorrent, 但是 udp 经常会被封锁, 使用 udp2raw 来做一个隧道. UDP2RAW 是绕过系统的协议栈, 把 UDP 数据包封装在一个伪造的 TCP 包头中发出, 直接用 IP 协议发出去, 但是不存在 TCP 复杂的机制 (例如握手), 因此依然是有 UDP 的高性能.

所以这个时候我的 TCP 的 qbit 是 tcp over udp (wireguard) over dummy tcp (udp2raw)

还遇到了 udp2raw 没法 dns 解析的问题, 顺便修改下 udp2raw 的源码, 让它支持 dns 解析

果然, 没过几天, wireguard 就相互连不上了… 这个之前用的时候也出现过

猜测是学校干的, udp 流量大了就给封了, 也不知道为什么要这样… 可以用一些工具, 使用类似隧道的方法, 把 udp 封装到 tcp 甚至 tls 中来发送, 例如 netcat. 但是可能由于配置等原因, 之前测试的性能很差, 准备使用一个 udp2raw 的工具. 这个工具的原理大概就是 udp 伪装成 tcp (但是不是 tcp), 直接用 IP 协议把这个包发出去, 所以保留了 udp 的性能.

感觉最好还是放到 Docker 里面, 这样可以方便运维迁移等, 可以给 udp2raw 单独做一个 docker, 已经有人做了

2.2 MTU 的计算

发现官方的 guide 建议的 openvpn 的 mtu 也是 1200…

重新梳理一下

数据 -> tcp -> ip -> wireguard 收到重新加密并且重新加上一个头 -> udp2raw 收到然后包装下 -> eth0的接口, 也就是 docker 的网络 -> wireguard -> udp2raw

仅仅使用 wireguard 时, 客户端的 TCP 看起来, 这个 MTU 是 1420, 于是数据加了实际的 TCP 头之后达到了 1420, 但是后面又加了 IP 层的头, Wirguard 的 UDP 头和控制校验之类的信息, 达到了 1500, 这一整个包作为一个 IP 包的内容放了进去. 所以这个 80 是 ip 头, UDP 头和 Wireguard 其他的信息

在仅仅使用 udp2raw 时, udp2raw 要求 UDP 头加数据应该达到 1375 的 TMU, 加了 udp2raw 的头和控制信息, 达到了真实的最大值 1500. 这个 125 也仅仅是 udp2raw 的 tcp 的头部和额外控制信息.

我们同时使用的时候, 就需要 80+125 的额外的数据, 就是下面这幅图, 因此我们设置 MTU < 1295

之前总会觉得很玄学, 但是实际研究发现, 其实原因并不复杂.

2.3 自动分片

但是 IP 层会自动分片, 并且在目的地自动组装, 这个除非设置了禁止分片. 为什么自动分片在上面错误发生时没有起作用呢?

wireguard 禁止分片 了, Wireguard本身会根据配置进行分片 (就是我们设置的 Wireguard 的 MTU).
但是我们为 Wireguard 设置了合适的 MTU, Wireguard 的分片策略不会影响最终效果. 最终 IP 层看到的是 udp2raw, udp2raw 也设置了禁止分片吗? 看了下似乎 是的, 因为 udp2raw 是直接绕过了 tcp, 发的 IP 包, IP 包里面设置了禁止分片. 因为 TCP 的包就是禁止分片的.

协议倾向于在自身层面控制数据包大小, 即主动设置 MTU, 而不是依赖 IP 层的分片机制. 原因可包括: IP 分片会增加网络传输的开销和重组失败的风险, 降低性能.

因此 udp2raw 禁止自动分片, 最终导致大于 MTU 的包都无法发送.
