udp2raw 的使用 | 增加DNS功能

发表于 2024-05-02 更新于 2024-10-06

最近使用 Wireguard 来利用公网跑 qbittorrent, 但是 udp 被学校封锁了, 使用 udp2raw 来做一个隧道. 这个是绕过系统的 TCP, 把 UDP 包一个 TCP 的头发出去, 但是 TCP 的握手这些不存在, 因此依然是有 UDP 的高性能.

所以这个时候我的 TCP 的 qbit 是 tcp over udp (wireguard) over dummy tcp (dup2raw)

还遇到了没法 dns 解析的问题, 顺便修改以下 udp2raw 的源码, 让它支持 dns 解析

1 wireguard 的隧道

果然, 没过几天, wireguard 就相互连不上了… 这个之前用的时候也出现过

猜测是学校干的, udp 流量大了就给封了, 也不知道为什么要这样… 可以用一些工具, 使用类似隧道的方法, 把 udp 封装到 tcp 甚至 tls 中来发送, 例如 netcat. 但是之前测试的性能很差, 所以使用了一个 udp2raw 的工具, 大概就是 udp 伪装成了 tcp, 但是不是 tcp, 所以保留了 udp 的性能, 具体实现没有看

感觉最好还是放到 Docker 里面, 这样可以关闭 docker 时候顺便一起关掉, 但是这样需要在 Docker 执行之前映射进去并且用脚本去执行; 或者直接自己做个 Docker, Wireguard + udp2raw; 也可以给 udp2raw 单独做一个 docker, 已经有人做了

但是这个 Docker 没有 arm 的, 就直接在主机跑吧

2.2 MTU 的计算

发现官方的 guide 建议的 openvpn 的 mtu 也是 1200…

重新梳理一下

数据 -> tcp -> ip -> wg0端口 -> wireguard重新加密并且重新加上一个头 -> udp2raw 加密然后包装下 -> eth0的接口, 也就是docker的网络 -> wireguard -> udp2raw

仅仅使用 wireguard 时, 客户端的 TCP 看起来, 这个 MTU 是 1420, 于是数据加了实际的 TCP 头之后达到了 1420, 但是后面又加了 IP 层的头, Wirguard 的 UDP 头和控制校验之类的信息, 达到了 1500, 所以这个 80 只是 ip 头, UDP 头和 Wireguard 其他的信息

在仅仅使用 udp2raw 时, 客户端应用认为 UDP 头加数据应该达到 1375 的 TMU, 加了 udp2raw 的头和控制信息, 达到了真实的最大值 1500. 这个 125 也仅仅是 udp2raw 的 tcp 的头部和额外控制信息.

我们同时使用的时候, 就需要 80+125 的额外的数据, 就是下面这幅图, 因此我们设置 MTU < 1295

之前总会觉得很玄学, 但是实际研究发现, 其实原因非常简单.

2.3 自动分片

但是IP层会自动分片, 并且在目的地自动组装, 这个除非设置了禁止分片. wireguard 禁止分片 了, Wireguard本身会根据配置进行分片 (就是我们设置的 Wireguard 的 MTU). 但是最终 IP 层看到的是 udp2raw, udp2raw 也设置了禁止分片吗?

看了以下似乎 是的, 因为 udp2raw 是直接绕过了 tcp, 发的 IP 包, IP 包里面设置了禁止分片. 因为 TCP 的包就是禁止分片的.
