ESPAsyncWebSrv 库源码

发表于2024-05-20|更新于2025-12-23|Embedded
|总字数:2.9k|阅读时长:12分钟|浏览量:

之前使用 Esp8266 做了一个控制床头灯的小玩具, 可以控制灯的开, 关, 闪烁和定时任务 (闹钟).
并且做了个 APP 来控制, 此外还让 GPT 写了一个网页端.

之前一直使用手机的 via 浏览器, 使用网页端设置定时任务. 但是突然有一天, via 浏览器无法设置了, 以为服务器坏掉了, 后来发现, 手机浏览器在开启电脑模式之后是可以正常设置的. 来查看一下到底是什么原因.

问题定位

这个最诡异的就是手机浏览器不可以正常使用, 但是手机浏览器开了电脑模式就可以.

ESP8266 代码

ESP8266 的 Arduino 代码很简单, 就是接收一个 post 请求, 然后根据请求里面的 定时的秒数, 以及定时 执行的任务. ESP8266 收到后会设置定时任务.

返回的不是一个 json 之类的, 可能不是很规范, 但是这是 GPT 写的 (甩锅). 里面就是拷贝过来 body, 然后用 json 解析.

手机抓包

首先抓个包看看是不是正常, 但是 via 浏览器似乎只能看到 request 的 url 和类型. 我们使用抓包工具来抓一下.

下面抓到的是 via 出现错误时候的包, 首先是请求设置定时任务, 之后是查询定时任务. 可以看到返回的结果是无法解析 Json

但是如果我的 via 浏览器开启了电脑模式就功能正常, 抓到下面的包. 可以看到只是 http requests headers 变了, 但是这次成功了.

串口输出

既然是 json 无法解析, 那么就来看一下串口输出的 post 的 body 吧. 连接串口之后, 每次手机模式下发送定时请求都会收到两个 body, 一个是 {"time":xxxxx,, 一个是 "status": "xxx"}.

重新抓包

这次选择抓取 ESP8266 收到的包.

在抓包之前首先看一下拓扑, 所有的设备都在一个局域网中, 只有其中的做 Web Server 的 Nginx 是在 Docker 中, 暴露端口 1314. Nginx 来回应静态网页, 对于 ESP8266 的 API 则是直接转发给 ESP8266.

这次我们使用 tcpdump 来抓包, 由于远端也有 1314 端口的调用, 所以我们使用 host 名字来抓包.

在 RK3399 上 (NGINX 所在的主机) 抓包手机和 ESP8266 的包.

这次一看, 手机上发的都是正常的包; via 浏览器是电脑模式时候包也是正常并且完整的. 但是在 via 浏览器没有开启电脑模式时候, 承载 HTTP 请求的 TCP 数据流被分割成了两个独立的报文段进行发送.

首先看到的是 4~5 确实是把 http request 的 tcp 分成了两个包 (TCP segment of a reassembled PDU), 因此下面返回了两个 Bad Request. 最后还出现了两个 RST, 这个可能是因为第二个 Bad Request 的回应返回太慢了, 在 FIN 之 后, 这个时候 TCP 已经要关闭了. 比如下面第二次尝试相同的操作, 两个 400 的 response 发送的快, 就执行了正常的四次挥手.

但是为什么会分包呢, 可以注意到上面 TCP 三次握手的时候, ESP 返回的 SYN-ACK 的包 (第二次握手), 设置了 MSS (Maximum Segment Size) 为 536B, 因此下面 NGINX 执行转发的时候, TCP 就自动分包了.

水落石出

很简单了, 就是 ESP8266 设置 MSS 为 536B.
如果是电脑模式, HTTP header 不是很长, TCP 不会拆分成两个; 但是手机模式, 有更长的 HTTP header, 恰好超过了 MSS, Body 分成了两片. 而我的代码并不太能处理两个 IP 包中的 TCP 消息.

这就是两个问题, 一是设置 MSS 为 536B. 但是真正的问题在于, ESPAsyncWebSrv 为节省内存没有组装数据, 而我的代码也没有处理, 而是在收到第一个数据块时就尝试解析完整的 JSON. 这个要看一下官方的代码和示例代码

搜索 ESP8266 MSS 536 会发现其实两种配置, 在 官方文档 中有说明.

重新配置就设置了更大的 MSS, 其实这里就已经解决了, 起码能用.

真正的原因

看起来能用的代码怎么会这样呢, 这个 AsyncWebServer 是基于 AsyncTCP 来实现的, 我们采用直接修改库源代码, 然后打 log 输出的方式来定位问题.

首先尝试在所有的 _parsedLength 变动的地方打 log, 查看输出, 主要的都是在中执行的处理, 计数也正常, 但是无法处理 Post 被分为两个包的情况.

回顾 Arduino 中的这段代码, 其结构如下:

看一下代码中这个的定义, 分别是 onRequest, onUpload, onBody.

而每次接收到 Post 的 Body (每个 TCP 包), 都会执行 if(_handler) _handler->handleBody(this, (uint8_t*)buf, len, _parsedLength, _contentLength); 上报一部分数据, 最终收到之后执行 _handler->handleRequest(this); 对整个 Request 进行处理.

最好的写法是参考官方示例, 但是我的是 Json 格式也懒得改了…

下面是几种解决办法:

  1. 从 Json ({"a":1,"b":2}) 的格式全部换成 Post 参数的格式 (a=1&b=2), 这个改的地方太多, 包括客户端和网页都需要变更.
  2. 应用级别设置缓存区, 收到部分 Post Data 就缓存起来, 如果 Request 就全部解析. 但是问题是需要设置 Ticker 来控制延迟, 如果定时没有结束, 就清空之前的缓存. 这样定时器还有两个, 需要设置一个 Pool 来管理, 并且 Ticker 的时间要设置好.
  3. 设置一个缓存 Request 的缓存, 使用 LRU 进行驱逐, 或者抽样驱逐超时请求, 而不是准确的定时驱逐.
  4. 修改源码, 提供存储 PostData 的选项, 就像解析 data 一样, socket 超时的同时释放资源.
  5. 直接摆烂, 反正现在 MTU 大了, MSS 大了就可以用了

我肯定选择5, 毕竟:
