实测Redis吞吐量 | redis-benchmark源码

发表于 2023-10-27

面试了两个小时, 中间甚至想逃跑一塌糊涂, 最后问了一个问题, Redis的吞吐可以是多少. 这个吞吐这些没有什么概念, 只知道Ray似乎能支撑上十万的并发, 来实际测一下.

有哪些Benchmark

首先来看一下redis有哪些Benchmark, 比如之前的gRPC, 记得官方就会给不同语言接口的性能基准测试; Ray本身也有, 不过好久没有看过了, 记得跑过两次都把Ray跑崩了… Redis应该也有. 简单搜索了一下.

首先就是官方的测试, 如果没有什么需求就用官方的这个啦, 官方用C写的应该是性能最高的了吧, 看了一下大概2k LoC, 这个安装Redis的时候应该会自带.

此外官方还提供了一个更专业的redis-benchmarks-specification, 这个包含不同语言的.

此外还有一些其他人写的, 比如用Python写的, 用Go写的等等.

我们一个一个来看&测试.

redis-benchmark

轻量应用服务器上的测试

实验是在腾讯云4C4G的轻量应用服务器(KVM over Intel Xeon Gold 6133 CPU @ 2.50GHz), 搭载了Redis server v=6.0.16.

首先使用自带的, 使用也很简单, 直接就可以用, 会执行一系列的操作, 记录throughput和latency. 我们先清空redis, 之后测试, 同时打开一个HTOP来进行观察.

htop的画风大概是只有两个再3%以上, Client占用97, redis占用83

得到一大串结果[点击下载]. 大概有下面的这些(GPT讲的)

其中除了RANGE的操作, 其他的都是再7~8w, 除了Range可能高一些(500元素的99%以上在3ms内), 其他的延迟99%以上都在1ms以内.

这个还可以多线程, 只要redis-benchmark --threads 3, 先清空, 再执行, 这时候Redis-server的cpu可以到达95以上, benchmark的三个cpu都在60以下. 最终的到的结果[点击下载]. 大部分都在8W附近.

几乎所有的性能都有一定提升, 除了最后的MSET(10keys)从78616.35 requests per second下降到了66622.25 requests per second, 猜测因为客户端争抢一些其他的资源什么的, 因为CPU太少了, 也没有局域网中的其他设备, 所以测试是非常非常不严谨的, 就是图一乐.

使用两个线程得到的结果就好的多[点击下载], 并且所有的都有提升, 一些甚至达到了十万的rps.

其实使用10个的pipeline就是会更多[点击下载], 达到70w附近

总结

他这个测试其实一般般, 我只测试了本身的吞吐, 没有测加上持久化的, 可以想象, 持久化他是专门的线程进行写入的, 应该不会影响太多, 但是在我的4c上的这个测试可能会争抢资源, 性能下降一点点.

感觉还是图一乐, 比如很多指令是同一个位置的写入, 可以说一直在cache里面, 无法给内存压力…比如pipeline感觉多次写入可能会被优化掉, 但是我也没看过源码…

话说回来, 大多数似乎都是这样测的, 毕竟测试是框架性能不是计算机的性能. 而且即便如此也还是很强的… 不使用测试的pipeline的情况下可以达到十万的吞吐, 开启之后能达到60w; 并且还是在性能如此受限的情况下, 基本上只有一个核, 还得抢一些其他的资源…

看了一眼GO和Python的, 其实也差不多, 但是感觉测出来会低一些, 毕竟是python.

此外也第一次算是看了redis源码, 比起其他的比如gRPC, 源码还是很紧凑易读的… 也领略了以下epoll的io复用, 确实能对性能带来极大的提升.

有机会看一下redis源码, 感觉写的还是满紧凑的.
