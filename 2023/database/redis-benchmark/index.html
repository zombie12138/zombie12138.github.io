实测 Redis 吞吐量 | redis-benchmark 源码

发表于2023-10-27|更新于2025-12-23|Database
|总字数:2.7k|阅读时长:10分钟|浏览量:

最近连续面试了两个小时, 面的一塌糊涂, 中间甚至想逃跑, 最后面试官问了一个问题, Redis 的吞吐可以是多少, 来实际测一下.

有哪些 Benchmark

首先来看一下 redis 有哪些 Benchmark, 比如之前的 gRPC, 记得官方就会给不同语言接口的性能基准测试. 这种组件大部分官方都会自带一份 Benchmark, Redis 应该也有. 简单搜索了一下:

首先就是 官方的测试, 如果没有什么额外需求就用官方的这个就够了, 官方用 C 写的, 看了一下大概 2k LoC, 这个安装 Redis 的时候应该会自带.

此外官方还提供了一个更专业的 redis-benchmarks-specification, 这个包含不同语言的.

此外还有一些其他人写的, 比如用 Python写的, 用 Go写的 等.

我们一个一个来看 & 测试.

redis-benchmark

轻量应用服务器上的测试

实验是在腾讯云 4C4G 的轻量应用服务器 (KVM over Intel Xeon Gold 6133 CPU @ 2.50GHz), 搭载了 Redis server v=6.0.16.

首先使用自带的, 使用也很简单, 直接就可以用, 会执行一系列的操作, 记录 throughput 和 latency. 我们先清空 redis, 之后测试, 同时打开一个 HTOP 来进行观察.

htop 显示 Client 占用 97, redis 占用 83

得到一大串结果 [点击下载]. 大概有下面的这些 (GPT讲的)

其中除了 RANGE 的操作, 其他的都是 7~8w, 除了 Range 可能高一些 (500 元素的 99% 以上在 3ms 内), 其他的延迟 99% 以上都在 1ms 以内.

这个还可以多线程, 只要 redis-benchmark --threads 3, 先清空, 再执行, 这时候 Redis-server 的 cpu 可以到达 95 以上, benchmark 的三个 cpu 都在 60 以下. 最终的到的结果 [点击下载]. 大部分都在 8W 附近.

几乎所有的性能都有一定提升, 除了最后的 MSET(10keys)从78616.35 requests per second 下降到了 66622.25 requests per second, 猜测因为客户端争抢一些其他的资源什么的, 本次测试环境受限, 非常不严谨.

使用两个线程得到的结果就好的多 [点击下载], 并且所有的都有提升, 一些甚至达到了十万的 qps.

其实使用 10 个的 pipeline 就是会更多 [点击下载], 达到 70w 附近

关键就是注册这个, ideal mode 用来测试耗费服务器的链接的, 所以我们正常测试都是进入 true 路径

c->context->fd 是之前获得的域套接字. 这里会在 fd 可写的时候触发 writehandler, 就是不停地写数据发指令. 当然这个是在后面的 aeMain 的循环中执行的.

通过 Unix 域套接字进行通信, 并借助 epoll 实现高效的 I/O 多路复用, 这是其实现高吞吐量的重要原因之一. epoll 它允许单个线程监视和处理成千上万个网络连接, 极大地提升了并发处理能力.

看 aeMain, 里面一个死循环, 持续调用 aeProcessEvents, 里面会 numevents = aeApiPoll(eventLoop, tvp);, 最终使用的是 epoll 系统调用:

之后处理这些事件, 一共 50 个客户端的 50 个事件, 分别调用下面

对于时间事件是后面单独处理的

关键就是我们的 writeHandler, 对于read也是类似的

writeHandler

之前注册的就是这个 writeHandler. 做了以下几件事情:

  1. 发送数据: 调用 write 系统调用，将客户端输出缓冲区 c->obuf 里的命令数据通过 fd 发送给 Redis 服务器.
  2. 状态翻转: 当所有数据都成功写入 (sdslen(c->obuf) == c->written)，writeHandler会做两件事：删除写事件监听和注册读事件监听.

readHandler

上面把 writeHandler 取消了, 那下次写入是什么时候呢? 来先看读取的处理.

也是类似的, 处理所有的 pending, 读取回复, 到最后会调用 clientDone, 默认是keepalive, 所以会执行 resetClint, 重新监听写入事件, 实现状态翻转.

freeClinet

每次benchmark执行完毕最后都要 freeClinet, 这个一般是 readHandler 调用 clientDone 调用的. writeHandler 也有检查, 但是我认为正常是 unreachable 的

总结

我只测试了本身的吞吐, 没有测加上持久化的, 可以想象, 持久化他是专门的线程进行写入的, 应该不会影响太多, 但是在我的 4c 上的这个测试可能会争抢资源, 性能下降一点点.

本文中测试非常不严谨, 比如很多指令是同一个位置的写入, 可能一直在cache里面, 无法给内存压力. 如果想要更好的进行测试, 可以 (最好还是有真实的线上负载, 以及超时等):

  1. 分离客户端与服务端;
  2. 使用随机键 (-r 参数) 模拟真实负载;
  3. 开启持久化来评估其性能开销.
  4. 一些其他的参数设置, 比如 client 数, pipeline 数.

看了一眼 GO 和 Python 的, 其实也差不多, 但是感觉测出来会低一些, 毕竟是 python, 循环的性能开销就会高很多.

此外也第一次算是看了 redis 源码, 比起其他的比如 gRPC, 源码还是很紧凑易读的. 也领略了以下 epoll 的 I/O 复用.
