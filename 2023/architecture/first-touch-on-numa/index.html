一次对 malloc 与 OpenMP 下 First-Touch 策略失效的失败追踪

发表于2023-09-17|更新于2025-12-23|Architecture
|浏览量:

最近看的 NUMA 绑核比较多, 来盘点一下 NUMA, 绑核, 以及 NUMA 系统中分配策略 (First Touch), 以及遇到的 First-Touch 失效的情况.

实验在 2Socket * 鲲鹏 920-Server (一共 128C 256G 4NUMA) 进行, 操作系统 CentOS Linux release 7.9.2009 (AltArch), 内核版本 4.14.0-115.el7a.0.1.aarch64, GCC 7.5.0, 开启了透明大页 (65536, 64KB), GNU libc 2.17

1 NUMA

一般的个人电脑都是 UMA (Uniform Memory Access), 不同的核心访问内存的速度是相同的. 但是在服务器 CPU 中, 可能有多个 CPU socket, CPU 访问另一个处理器上的内存会更慢一些, 但是访问本地的内存会快一些， 这就是NUMA (Non-Uniform Memory Access). 有些时候, 为了提高单个处理器的带宽, 满足不断增长的核心数的带宽需求, 也会把单个处理器上的核心分为几组, 每一组是一个 NUMA Node, 拥有自己的内存控制器. 核心自己所在的 Node 叫 Local Node; 其他的 Node 就是 Remote Node, 访存延迟和带宽性能都会有所下降; 相同 Socket 不同的 Node 称为 Neighbour Node， 速度介于两者之间.

比如下图的鲲鹏 920-Server, 每一个 SoC 拥有两个 CPU-Compute Die, 每一个 CPU Compute Die 拥有两对 (4个) DDR 控制器, 一个 SoC 有 8 个 DDR 控制器, 可以达到 200 GB/s 的带宽. 如图所示, 2 个鲲鹏 SoC 就会有 4 个 NUMA Node.

一些 CPU 也可以开启或者关闭 NUMA, 甚至可以设置 NUMA Node 的数量. 可以使用 numactl -H 或者 lscpu 来查看是否存在 NUMA, 比如在鲲鹏 920 上的输出如下, 里面还有 NUMA Node 间的 Distance. 这个 Distance 一般是直接从 ACPI (Advanced Configuration and Power Interface) SLIT (System Locality Information Table) 读取出来的. 可以看到, 即便都是跨 Socket, Distance 也可能是不同的.

NUMA 带来的好处就是可拓展性, 内存带宽跟随 CPU 核心数量上涨.

但是这也带来很多挑战. 首先一个最直接的就是, 程序很可能访问远端的内存, 这时候内存的速度就会下降很多. 另一方面, 可能会导致这种 NUMA 可能导致很多 CPU 争用一个内存控制器, 或者加大通讯压力等. 最后伪共享这些可能进一步加剧性能下降.

解决的办法也有很多, 可以通过 NUMA 感知的内存分配, 绑定核心 (taskset, OpenMP proc bind, numactl 等工具), 异步通讯, NUMA 感知的算法等等来进行解决.

2 Linux 的 NUMA 内存分配策略以及 FirstTouch 失效问题

这里说的比较乱, 最好还是去看 Linux Kernel Documentation

可以通过设置内核参数, 比如 zone_reclaim_mode, 设置为 0 则内存不足的时候在远端的 CPU 分配内存; 也可以使用一些工具或者库, 比如 numactl, 比如 numactl --interleave=all; 还可以使用 libnuma 在编程中手动指定分配位置.

numactl 的绑核有下面几种类型: default 在本地 numa node 上分配内存; bind 指定在一系列特定的 numa node 上分配内存; interleave 在一系列 numa node 上分配交错内存; preferred 指定优先选取的 node

但是 Linux 系统的分配策略不是在申请内存的时候就生效的. Linux 内存分配策略就是 First Touch, 第一次访问内存页 (一般 4K) 的时候, 会产生缺页中断, 之后才会分配物理内存, 并且分配给一个物理页框. 在 NUMA 的系统中, First Touch 分配的内存页在第一次访问这个内存页的核心所在 Node.

2.1 OpenMP 向量加法 - 一个线程来初始化

来手动写一个 OpenMP 的加法.

这个程序先申请 3 个 2 GB 的数组, a 和 b 然后用 1 个 OpenMP 的线程进行初始化, 之后使用 OpenMP 开启 128 个线程去做向量加法计算结果保存至数组 c, 打印出来向量加法部分的时间.

输出 OpenMP 初始化和向量加法的线程和物理核心的映射关系, 同时还会输出该线程处理的数据内存第一个 Page 所在的 NUMA Node.

OpenMP 的 for 会把 for 循环任务按照循环变量切分成 128 份, 从第一份到最后一份依次分配给 0 ~ 127 号线程, 但是线程执行了在哪个 CPU 是调度决定的.

使用 gcc -o sum -O0 -pthread -lnuma -fopenmp sum.c 编译, 执行. 注意, 上面的初始化是由一个线程完成的, 但是执行向量加法的操作是 128 的线程一起执行. 最终加法部分用时是 226ms

2.2 128 个线程来初始化

按照我们之前的说的 First Touch 来分析, 由于初始化由一个线程执行, 该线程被分配至 62 号 Core 执行, 所以基本上全在 1 号 Node.

但是如果使用 128 个线程来进行数组的初始化, 由于这些线程被分配到不同 Node 的 Core, 内存页也会分不到不同的 Node. 只需要把初始化的线程数改成 128, 会来到 173ms 附近.

向量加法是一个计算很稀疏的操作, 是 memory bound 的. 之前只能用到一个 NUMA Node 的内存的带宽, 现在可以用到所有的 NUMA 内存的带宽.

最终, 程序均匀取了内存 128 个地址, 观察所在 node: ./sum | grep node | awk '{print $11}' | sort | uniq -c

本来预期是每个 Node 上内存 均匀分配, 但是最终得到的结果却出人意外, 在 Node 1 上分配最多.

OpenMP 对于循环的默认处理是直接切分为 128 段, 分配给 128 线程, 每个线程处理自己被分配的部分. 如果严格按照 First Touch 分配, 理论上应该是在 4 个 Node 上均匀分布, 为什么分配的位置并不均匀呢?

4 核心亲和性

之前说过, 一种常见的应对 NUMA 的策略就是绑核. 我们设置一下打印出来多线程的核心位置, 然后编译 gcc -o print_cpu_id -O0 -pthread -lnuma -fopenmp print_cpu_id.c

./print_cpu_id 1 打印主线程, ./print_cpu_id 128 打印128个线程, 可以看到, 输出的结果非常多, ./print_cpu_id 128 | awk '{print $7}' | sort | uniq | wc 输出一共 124 个核心, 可能因为线程的调度, 所以刚好有些线程执行到查询 Core 时候恰好在同一个核心, TOP 命令上面看到的其实是 128 Core 都用到了.

4.1 taskset 和 numactl

taskset 是一个最简单的绑核的工具. taskset -c 1 ./print_cpu_id 1 打印出来核心在 001, 多线程也一样 taskset -c 1 ./print_cpu_id 12, 也可以 taskset -c 0-11 ./print_cpu_id 12

我们试一下 strace 一下 strace taskset -c 4 ./print_cpu_id 1, 发现多了一个 syscall, sched_setaffinity(0, 256, [4, ...]) = 0. 其实这个我们自己也可以做.

当然可以使用 numactl 设置亲和性, 比如 numactl -C coreid.

此外 numactl 还可以设置内存分配策略, 比如 numactl --interleave=all 在不同的node之间分配内存. 拿来测试一下之前的向量加法 numactl --interleave=all ./sum | grep node | awk '{print $11}' | sort | uniq -c, 修改代码使用一个线程来初始化, 按照之前的 FirstTouch, 应该是都在一个 Node 上, 但是结果如下, 内存分布在不同的 node 中, 效果非常好, 效果大概是内存分成了 5 个部分, 开头和结尾在一个 Node, 中间三个段在一个 Node. 用时是 181ms, 虽然内存在 Node 间分布更均匀, 但是比 128 线程初始化来 First Touch 的效果 (173ms) 稍微差一点, 猜测可能是过多访问远端节点造成的.

4.3 OpenMP 和 MPI 绑定核心

手动设置亲和性是代价比较高的, 而且很不方便, 但是如果是 OpenMP 和 MPI 程序来说 taskset 和 numactl 依然不够专业和方便, 这些框架提供了自己的绑核机制.

4.3.1 MPI

使用 mpicc -o mpi mpic.c 编译, 出来之后 mpirun -np 128 ./mpi | awk '{print $10}' | sort | uniq | wc, 开启 128 个线程, 合并相同的 cpuid, 然后发现有 86 个 cpuid.

mpi 的绑核可以通过 mpirun 时候指定给出, --bind-to <policy>, 策略可以是 none, hwthread, core, l1cache, l2cache, l3cache, socket, numa, board, 默认绑定到 core 上, 比如 mpirun -np 128 ./mpi | awk '{print $10}' | sort | uniq | wc, 得到 128 个核心都有绑定线程, 但是映射是随机的, 如果绑定 l3, numa 或者 board 就和之前一样了.

也可以和之前一样. --report-bindings 设置打印绑定内容, mpirun -np 4 --report-bindings --bind-to socket ./mpi 执行结果太长了不展示了.

可以尝试 strace 一下, 但是由于开启的这个进程只负责屏幕输出, 启动之类的, 所以需要追踪 fork, 但是 strace -ff -o trace mpirun -np 4 --report-bindings --bind-to socket ./mpi 会出现段错误. 同时有太多进程, 会创建大量的 trace.$pid, 因此可以创建一个 warpper 的脚本

之后设置成可执行, 然后 mpirun -np 4 --report-bindings --bind-to socket ./mpi, 看输出的 stace, 依然没有 sched_setaffinity, 猜测可能是创建之前就设置好了. 但是从 sched_getaffinity 结果来看, 其亲和性就被设置成第一个 socket 所有的 core (0 ~ 63).

上面的这个是绑定策略, 但是线程和核心的映射依然是随机的. 想要控制内存映射需要使用 --map-by <arg>, 参数可以是 slot, hwthread, core, L1cache, L2cache, L3cache, socket, numa, board, node, sequential, distance, and ppr.

mpirun -np 8 --bind-to core --map-by node ./mpi, 每个 NUMA 的线程 ID 分别为 0 4, 1 5, 2 6, 3 7, 并且绑定每个核心.

4.3.2 OpenMP

OpenMP 的核心绑定其实也很简单. 首先 OpenMP 的线程其实也只是启动一次, 如果手动绑定线程的分配手动配置亲和性其实就可以了, 但是猜测这个可能会影响调度, 并且也会有一点点的开销. 最好就是使用 OpenMP 的核心绑定.

OpenMP 可以使用环境变量或者 pragma 来配置. MPI 是轮询地把进程分配到 map-by 的集合上, 然后绑定到该集合上面的某个 bind-to 子集上, 其实是相当于 OpenMP 的 Spread.

OpenMP 首先分配了多个 PLACES, 可以是线程, 核心, socket, 可以使用 OMP_PLACES 来进行指定.
之后用 OMP_PROC_BIND 指定分配策略, 有 spread, close, master, true, numa_domains, ll_caches. Spread 就是类似 MPI 的轮询分配线程的方式, 每个线程轮询绑定下一个 PLACES 的 core 上; Close 是依次分配, 分配完一个区域再分配下一个区域; master 就是使用主线程来执行. PLACES.

下图是一个 4C8T 的多线程, OMP_PLACES 设置为 core, 那么每个 Core 有 两个线程, 使用 spread 和 close 其线程和核心的映射分布如下 (方框表示 OpenMP Places，数字表示线程 ID):

如果想要更细粒度的绑定, 比如根据 LLC 之类的, 可以自己配置 OMP_PLACES， 比如 OMP_PLACES="{0},{1},{2},{3}" 表示 4 个 PLACE, 每一个 PLACE 是一个线程; OMP_PLACES="{0:2}:32:8" 表示, 开始是 0:2, 0 开始的 2 个线程, 就是 0 和 1, 32 表示有 32 个 Places, 就是一共使用了 64 线程, 8 表示间隔, 第一个 PLACE 是 0 和 1, 第二个是 8 和 9, 以此类推. 也可以组合使用, 比如 export OMP_PLACES=“{0}:4:1,{78}:4:1”, 定义了 8个 Place, 分别是 0, 1, 2, 3, 78, 79, 80, 81 号线程.

5.3 单个页面上手动验证 FirstTouch

哪到底是 glibc 的哪些行为导致了 first touch 提前生效呢? 甚至开始怀疑 First touch 是不是以页面为单位的 (虽然文档里这么讲的).

写一个 pthread 的程序, 在 Node3 上执行主线程, mmap 分配内存, 然后去触碰中间一个页面的开头的区域; 然后其他的线程在 Node0 上去触碰所有的内存, 查看一下内存的分布情况.

结果就是只有 512 是在 3 号 Node上, 是以页面为单位. First touch 这里表现没问题.

改成 malloc, 变成了 0, 512, 513 是在 Node3 上, 这个和上面代码分析以及 FirstTouch 表现的完全一样. malloc 会 touch 开头, 并且偏置导致对 512 这部分内存一部分偏移到 513, 因此 3 块内存都是分配在 Node3.

问题到底在哪里? 已知:

  • malloc 过程中只碰前 16B
  • mmap 完全不碰
  • first touch 是以页面为单位的
  • malloc/mmap + pthread 在两个线程时候完全正常.
  • malloc + OpenMP 开头和结尾会被分配到主线程所在 node
  • mmap + OpenMP 开头和结尾是正常的

那就是 malloc + OpenMP 会出现一些怪东西, 触发了 First touch. 是OpenMP + glibc 出现了这个问题?

这就 watch 一下这一段内存, 从之前的分析, malloc 本身是正常的, 那就可以 malloc 之后再监视最后的一段内存.

使用 gdb 进行调试. 设置 Line 18 一个断点, 执行运行 (获取到 malloc 地址后暂停), 之后设置监视 0x400080330000 位置的 16 字节数据, 继续执行:

试了试 watch 和 rwatch (按说读取不会触发) 都没看到访问最后的那一个页面. 如果设置成 65536 字节的 wathch 速度又会非常非常慢.

想到一些通过设置权限来捕捉内存访问, 那就把a整个数组全都设置成不可读, 不可写 call mprotect(0x400000340000, 2147483648, 0)

continue 捕捉到的是 a 再赋值的时候, call mprotect(0x400080340000, 65536, 0) 看一下最后一个页面, 发现 info threads 显示就是第128个线程访问的.

后面 rw 是正常, malloc 不会 rw 访问后面的这些应用.

也就是说, malloc 到程序执行结束, 第一次访问数组 a 中最后地址的是最后一个线程, 但是内存确被分配到了 Node 0.

5.3 诡异到放弃: 只要头部添加偏置, 则 FirstTouch 不符合预期

手动模拟 malloc 进行 mmap 分配也会观察到类似的 malloc 相同特征, 本来规规整整的 First touch, 突然内存尾部多了好多分配在主节点的内存

并且, 如果使用 mmap 分配规整的一大块内存, 使用 128 线程来初始化是正常的, 但是使用 4 线程初始化也会出现类似更加严重的现象

所以怀疑是内存分配策略做了一些优化之类的. 测试到发现更诡异了

6 吐槽

总是写得非常长, 太冗余, 其实本来想写一下如何让 vector 不初始化来使用, 以及如何调用私有函数这些.

但是写的时候介绍背景就已经很长了, 想先写 First Touch 来配置内存分布, 然后 Bind Core 的优化.

就这样写到一半的时候发现 First touch 不是总是起作用, 猜测是 glibc 在内存做了一些标注; 之后查看 syscall 什么也没发现; 然后又去看 malloc 的 mmap 的实现, 发现只会影响第一个页面.

后面的页面 NUMA 的分配更诡异, 使用 mmap 直接分配 (页面开头对齐每个线程开始写入的地址), 则 FirstTouch 符合预期, 如果使用 malloc 分配, 或者手动添加 16B 偏置, 则 FirstTouch 总是失效, 这使得问题的根本原因更加扑朔迷离, 也没有找到相关的问题讨论或者文档说明.

最终结果令人沮丧, 最终也没有探究明白为何 FirstTouch 会这样的表现, 怀疑是数据预取的问题. 但是后续没有用 Perf, 或者一些其他专业的方法去调整测试.

题外话: 之前使用 lmbench 测试访存的延迟, 得到内存工作集在 128KB (2 倍的 L1 缓存) 时候, 延迟才下降到 L2. 但是后来更新了系统, Ubuntu22, 就是正常的 (64KB 出现延迟的突变). 两者 BIOS 都开启了数据预取, 这个我现在想不到是什么原因导致的, 并且感觉可能和现在的状况有关. 但是现在新版本系统没有测试我的这些代码.
