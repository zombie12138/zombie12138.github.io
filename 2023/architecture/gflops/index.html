CPU 浮点数运算速度理论上限 | 如何计算 GFLOPS

发表于2023-10-16|更新于2025-12-23|Architecture
|浏览量:

最近看了一些 ARM 测评的文章, 注意到鲲鹏 920-6426 的浮点计算性能似乎有很多不同的说法, 来盘点一下到底怎么计算和测量一个 CPU 的 GFLOPS.

实验在 2Socket * 鲲鹏 920-Server (一共 128C 256G 4NUMA) 进行, 操作系统 CentOS Linux release 7.9.2009 (AltArch), 内核版本 4.14.0-115.el7a.0.1.aarch64, GCC 7.5.0, 开启了透明大页 (65536, 64KB), (GNU libc) 2.17

1 问题的开始

1.1 665 Tflops 的鲲鹏

由于鲲鹏本身不是为了 HPC 设计的, NEON 矢量指令集又只有 128 bit, 浮点数比较孱弱, 但是其具备 多少 GFLOP, 不同论文有着完全不同的说法.

首先是参考文献[1]中提到, 如下图所示, 给出的结果是单精度的峰值是 `1.331 TFlop/s`, 双精度峰值是 `665 TFlop/s`, 标量的计算峰值是 `332 Gflops`. 这个结果显然是不正确的, 至少单位写错了. 我们先对这个 GFLOPS 有个简单的认识, G 和 T 都是国际单位制前缀, 分别是 `10^9` 和 `10^12`. 先来看看宇宙最强计算卡 H100 的 FP 算力[2]. 鲲鹏 920 只是 64C64T 的 ARM-v8 的 处理器, 这里是有明显的 TYPO. 作者可能是少打一个小数点, 导致鲲鹏性能提升了 1000 倍. 但是这个数据 (665 Gflops 和 1.33 TFlops) 依然是不准确的, 我们先来看看其他出处是怎么说的. ### *1.2* 1,33 TFlops 另一篇来自俄国有机化学实验室的文章[3]中提到 (听起来怪怪的), 鲲鹏 920 的 *Peak performance(DP),TFLOPS* 是 `1,33`, 我理解逗号是小数点, 即便如此, 依然还要比第一篇文章中想要表达的数据快了一倍 ### *1.3* 最靠谱的数据 上海交大在 ARM HPC Asia 2019 的一次演讲中提到[4], 鲲鹏 920 的浮点数计算能力如下: SP 是单精度, DP 是双精度. 但是这个数据似乎是他们实际测量出来的, 而不是理论计算, 这个数据应该是相当真实准确的. ### *1.4* 社区的答案 在社区中, 也同样有一点讨论, 华为云社区有人给出了公式如下[5]: ## *2* 鲲鹏 920 的 GFLOPS 计算和测量 但是到底哪一个说法才是正确的呢? ### *2.1* Gflops 的计算 首先是 GFLOPS 的理论计算, 其实华为云社区的回答是正确的. 但是我们还需要拿到鲲鹏的 FMA (ASIMD FP multiply accumulate) 指令的吞吐量, 翻了一下手册, 其他的大部分都是每周期两个, 但是没有找到 Taishan V110 的确切的数字. 在 WikiChip[6] 上说每周期执行 **单个双精度 FMA** 或者 **两个单精度 FMA**, 但是之前的上交演讲中也支持了这个说法. 其次就是 FMA, FMA 到底算一条还是两条呢? 我看了一些论坛上的讨论[7][8]都是给出来的按 2 来计算, NVIDIA H100[2]也是按 2 来计算的 (`1.980GHz*132SM*128FP32/SM*2(FMA)=66908Gflops`). 所以理论上单精度是 `(128bit/32bit)*2.6GHz*2(FMA per clock)*(1+1)*64=2662.4 Gflops`, 双精度只有数据变成了 64bit, 每周期指令是 1, 是单精度浮点运算速度的 1/4, 是 `665.6 GFLOPS`. ### *2.2* 实际测试验证 #### *2.2.1* 单核 当然我们似乎得到了这个数值, 我们还需要手动验证一下, 可以手动写一些汇编, 来检验整个的吞吐. 看到了有一些这样的工具存在[10], 但是感觉比较复杂, 一个最最简单的就可以了, 最终参考了一个仓库来写[9]. 首先是汇编部分, `fmla` 指令就是把后面两个向量相乘, 累加到第一个向量. 需要构造 16 条流水线, 指令之间没有数据依赖. double 的话每条指令处理两组浮点数的 fmla, 一条指令是 4 次浮点运算, 循环一次就是 4*16 就是 64 个浮点运算

然后写一个 c 来调用这段汇编 然后直接 gcc 编译, taskset 到 core 0 运行, 得到的结果为: 单核性能是 10.4 GFLOPS, 所以, 一个 Kunpeng 920-6426 的双精度浮点计算的性能是 665.6 GFLOPS, 就是我们的理论上限 `665.6 GFLOPS`. 测一下单精度, 汇编寄存器改成 `4s` (即四位单精度浮点数), 计算的过程每一轮循环是 `128` 次单精度的计算. 得到SP的单核性能是 `41.583994GFLOPS` 多核心就是 `2661.4GFLOPS` 非常接近 `2662.4GFLOPS` #### *2.2.2* 多核 加上一层 OpenMP 的 `#pragma omp parallel`, 先预热, 再进行计时. 这里测试了搭载两片鲲鹏处理器的机器, 最终得到双精度计算为 `1271.590910 GFLOPS`, 就是说单个 SOC 达到了 `635.795455 GFLOPS` 以上 (包含一些等待和同步的时间), 非常接近上限 `665.6 GFLOPS` 单精度是达到了 `5085.4412485 GFLOPS`, 单个 SOC 达到了 `2542.72062425GFLOPS`以上, 非常接近 `2662GFLOPS` 对于 `fmul` 和 `fadd` 吞吐也是和 `fmla` 完全一样的, 单双精度吞吐分别 1331 GFLOPS 和 332.7 GFLOPS 如果使用标量的话, 单双精度吞吐都是 2 IPC (Instruction Per Cycle), 也就是都是 332.7 附近. ## *3* AMD EPYC 7302 那么鲲鹏这款处理器浮点数运算到底有多弱呢? 我们对比下 AMD EPYC 7302, 两者发布时间比较接近. 这个先测试单线程, 过程类似, 只是需要换成 avx256 指令集: 每一次循环 `32 * 2 * 4` 的浮点计算, 修改一下 `.c`, 可以得到单核性能, `52.5 GFLOPS`. 但是由于超线程两个线程共享FPU, 所以 ` NUM_THREADS=2 OMP_PROC_BIND=close ./fp` 两个线程执行还是共享的FPU, 所以其吞吐依然是 `24.73*2`, 甚至性能还下降了一点点 所以 16 个核心就是 `840 GFLOPS`, 16c 的处理器双精度浮点数性能已经大于鲲鹏 920. 对于 SP (Single Precision), 得到的结果是超线程是 `49.5`, 两个线程就是, 一个核心提供了接近` 100` 的单精度, 但是对于单线程 70 的结果还是明显提升了, 怀疑是有一些其他的资源限制, 使用两个线程缓解了这个限制, 然后瓶颈转移到了浮点处理单元上 (Intel 的结果类似) 测了一下, 浮点数的乘法 (vmulpd) 也是类似的, 超线程就能到达 (2 IPC * 4 F/I * 3.3 GHz). 加法 (vaddpd) 的结果也是和加法完全一样 ## *4* 总结 我们从两篇论文, 社区和一个演讲中得到了鲲鹏的不同的浮点数计算的 GFLOPS. 进行简单的计算之后得到了真实的理论上限, 双精度 DP 是 665.6 GFLOPS, 单精度 SP 是双精度的四倍, 是 2662.4 GFLOPS. 并且最终使用汇编来验证了这个结果. 回头看, 一个简单的问题两个论文给出了错误的答案. 第一篇论文双精度写对了, 但是单精度结果小了一半, 并且存在明显的 TYPO. 第二篇论文双精度 GFLOPS 写成了理论的两倍, 可能没考虑到鲲鹏的 FMLA 单周期只能发射一条. 第三个上海交通大学的这个非常准确, 而本次的实测单核性能比其报告的值更高, 更接近理论上限; 最后社区的结果给的也很准确. 还有就是 Wiki 目前没看到任何错误, 比一般论文还要靠谱. 顺便测了一下吞吐. 结果如下 (纯 MD 似乎不支持合并单元格, 但是写 htm 的表格太麻烦了) ## 参考文献
