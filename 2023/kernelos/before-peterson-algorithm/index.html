Peterson 算法前置篇 -- 在 So 中互斥访问变量

发表于2023-08-03|更新于2025-12-23|Kernel&OS
|浏览量:

之前面试某公司的暑期实习, 聊到共享库的时候, 面试官问了一个问题:

我们现在遇到一个问题, 在动态链接库中共享一个全局变量, 我们想要互斥访问这个全局变量, 应该怎么做?

自己想的肯定是互斥加锁就可以了, 肯定有对应的系统调用. 但是面试官追问 如果没有现成的互斥锁应该如何访问共享的变量? 只记得操作系统课程中讲过软件实现互斥锁的算法,
但是忘记了算法的具体实现, 甚至忘记了算法名字…

这个可以拆成几个问题

  1. So 中如何共享一个变量
  2. 有哪些机制可以实现进程间互斥访问 so 中的变量
  3. Peterson 算法的实现

这个问题比较基础 (虽然回答的一塌糊涂), 但是可以学习的地方有很多, 这一章先来讨论在 so 中如何互斥地共享某个变量, 即多进程间的互斥问题.

所有的代码参见我的 仓库, 使用的是腾讯云4C4G的轻量应用服务器 (KVM over Intel Xeon Gold 6133)

1 So 的全局变量是共享的吗？

其实也说不上共享, 是 Copy On Write. 这里可以做个小测试, 两个进程共享一个共享库, 两个进程去分别初始化和读取动态库的全局变量, 如果 可以读取到对方的写入 就说明是共享的, 否则就不是.

最终的结果是第一个 B.print 什么都没有输出, 即 A 的初始化对 B 无效; 后面 A 和 B 的输出也都是自己本身的内容.

这个问题的设定, 在实际的操作系统机制下可能与直觉有所不同. 由于 COR (Copy-On-Write) 的存在, 动态库中的全局变量并不会在多进程间直接共享

如果有能力可能会继续剖析动态链接器在这方面的实现.

在尝试从零开始用纯软件构建互斥算法 (如 Peterson 算法) 之前, 了解并评测当前系统中已有的同步原语是至关重要的一步. 这不仅能为我们提供一个性能基准, 也能帮助我们理解这些系统调用背后所做的优化. 接下来，我们将分别使用信号量, 互斥锁, 原子操作和自旋锁来解决这个计数问题, 并粗略比较它们的性能.

3 互斥访问

3.1 使用信号量

这个其实就是多进程中的加锁, 可以直接使用信号量或者锁来进行进行互斥访问

可以使用 Linux 的 <semaphore.h> 中的 sem_t 来进行互斥访问, 顺便把 wakeup 和 finished 也用信号量来进行实现的 (语义会明确些).

3.2 更轻量的mutex锁

使用信号量是比较重的操作, 每次都要切换内核态和用户态, 所以性能下降如此严重. 这里如直接使用多进程的 mutex, mutex 的功能相比信号量更少, 针对互斥访问的场景, 会有一些额外的优化.

提速的原因是因为他会先尝试获取用户态的锁 (先 spin), 获取成功就会直接进入, 如果不成功才会使用 futex 这些系统调用.

3.3 使用原子操作

直接使用 atomic add 来原子加法, 估计会比之前的速度都要更快一些.

可以直接把 mutex 变量相关的删掉, 把 int 换成 std::atomic_int

3.4 自旋锁

原子操作是利用硬件指令集直接实现的, 效率很高, 但它更像是一个针对特定操作的特例化方案. 有些数据不方便使用原子操作, 就可以用自旋锁, 比较轻量, 在锁竞争不激烈且临界区执行时间极短的场景下, 自旋锁避免了线程调度的开销, 通常被认为是性能最高的锁之一.

令人意外的是, 本次测试中自旋锁的耗时 (0.17s) 甚至低于原子操作 (0.24s).这个现象值得深入探究, 可能与 std::atomic 默认的内存序带来的额外开销有关. atomic 默认的内存序是 memory_order_seq_cst (顺序一致性), 这是最强的内存序, 开销较大. pthread_spinlock 的实现可能使用了更宽松的内存序或不同的底层机制, 导致在特定测试场景下开销更小.

4 总结

多进程访问 so 中的 global variable 不会共享, 只能使用 mmap 来映射一段共享内存. 互斥访问就是多进程中的互斥, 分别使用了信号量 (1.87s), 互斥锁 (0.94s), 原子操作 (0.24s), 自旋锁 (0.17s) 解决了问题, 但是依然有很多疑问:

  • 这些调用背后做了哪些工作, 为什么差别这么大? 自旋锁为什么比原子操作还要快.
  • 还有就是最初的问题还是没有解决, 就是如何实现 Peterson 算法, 如何在现代计算机中如何实现 Peterson 算法?
