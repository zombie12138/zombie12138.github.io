CPU浮点数运算速度理论上限 | 如何计算GFLOPS

发表于 2023-10-16 更新于 2024-01-17

最近看了一些ARM测评的文章, 注意到鲲鹏920-6426的浮点计算性能似乎有很多不同的说法, 关于GFLOPS的计算不同的论文也是有不同的说法. 来盘点一下到底怎么计算一个CPU的GFLOPS.

实验在2Socket * 鲲鹏920-Server(一共128C 256G 4NUMA)进行, 操作系统CentOS Linux release 7.9.2009 (AltArch), 内核版本4.14.0-115.el7a.0.1.aarch64, GCC 7.5.0, 开启了Page Size(65536, 64KB), (GNU libc) 2.17

问题的开始

665Tflops的鲲鹏

由于鲲鹏本身不是为了HPC设计的, NEON又只有128bit, 所以浮点数比较孱弱, 但是论文中的GFLOPS似乎都是会有不同的说法.

首先是参考文献[1]中提到, 如下图所示, 给出的结果是单精度的峰值是1.331 TFlop/s, 双精度峰值是665 TFlop/s, 标量的计算峰值是332 Gflops.

首先就是这个单位, FLOPS是floating point operations per second, 是每秒计算浮点数的数量, 用来衡量浮点数计算的峰值, 可以写成FLOPS.

这个结果太假了, 鲲鹏是64C64T的ARM-v8的CPU. 我们先对这个GFLOPS有个简单的认识, G和T都是国际单位制前缀, 分别是10^9和10^12. 先来看看宇宙最强计算卡H100的FP算力[2].

首先鲲鹏的双精度计算竟然比单精度计算速度还要高, 而且强大如H100双精度计算也比不过文章中的鲲鹏, 只能说瑶瑶领先.

认真分析一下, 作者可能是少打一个小数点, 导致鲲鹏性能提升了1000倍, 这样就可以解释了, 单精度是双精度速度的两倍. 但是这个数据(665Gflops和1.33TFlops)依然是不准确的, 我们先来看看其他人是怎么说的.

1,33TFlops?

另一篇来自俄国有机化学实验室的文章[3]中提到(听起来怪怪的), 鲲鹏920的Peak performance(DP),TFLOPS是1,33, 我理解逗号是小数点, 否则又要瑶瑶领先了. 即便如此, 依然还要比第一篇文章中想要表达的数据快了一倍

实际的计算

但是到底哪一个说法才是正确的呢?

Gflops的计算

首先是Gflops的计算, 其实华为云社区的回答其实是比较正确的.

但是我们还需要拿到鲲鹏的FMA(ASIMD FP multiply accumulate)指令的吞吐量, 翻了一下手册, 其他的大部分都是每周期两个, 但是没有找到Taishan V110的确切的数字, 在WikiChip[6]上说每周期执行单个双精度FMA或者两个单精度FMA, 但是之前的上交演讲中也支持了这个说法.

其次就是FMA, FMA到底算一条还是两条呢? 我看了一些论坛上的讨论[7][8]都是给出来的按2来计算(毕竟是乘法和加法, 当然厂商喜欢这样来计算), NVIDIA H100[2]也是按2来计算的(1.980GHz*132SM*128FP32/SM*2(FMA)=66908Gflops).

所以理论上单精度是(128bit/32bit)*2.6GHz*2(FMA per clock)*(1+1)*64=2662.4 Gflops, 双精度只有数据变成了64bit, 每周期指令是1, 所以双精度是665.6Gflops.

实际测试验证

单核

当然我们似乎得到了这个数值, 我们还需要手动验证一下, 可以手动写一些汇编, 来检验整个的吞吐.

首先是汇编部分, 就是把后面两个向量相乘, 累加到第一个向量. 需要构造16条流水线, 没有数据依赖. double的话每条指令处理两组浮点数的fmla, 一条指令是4次浮点运算, 循环一次就是4*16就是64个浮点运算

多核

加上一层OpenMP的#pragma omp parallel, 先预热, 再进行计时.

最终得到双精度计算为1271.590910GFLOPS, 就是说单个SOC达到了635.795455GFLOPS以上(包含一些等待和同步的时间), 非常接近上限665.6GFLOPS

单精度也是5085.4412485GFLOPS, 单个SOC达到了2542.72062425GFLOPS以上, 非常接近2662GFLOPS

对于fmul和fadd吞吐也是和fmla完全一样的, 也就是说如果乘法和加法不均衡, 无法使用fmla的话,会少一半FLOPS, 单双精度吞吐分别1331和332.7

如果使用标量的话, 单双精度吞吐都是2 IPC, 也就是都是332.7附件

总结

我们从两篇论文, 社区和一个演讲中得到了鲲鹏的不同的浮点数计算的强度. 进行简单的计算之后得到了真实的理论上限, 双精度DP是665.6GFLOPS, 单精度SP是双精度的四倍, 是2662.4GFLOPS. 并且最终使用汇编来验证了这个结果.

回头看, 第一篇论文全都写错了, 单位这种低级错误就不说了, 双精度写对了, 但是单精度写小了一半. 第二篇论文双精度GFLOPS写成了理论的两倍, 可能没考虑到鲲鹏的FMLA单周期只能发射一条; 第三个上海交通大学的这个满准确, 但是我测量的单核性能比他的高了不少, 更接近理论上限, 多核他应该也是手动测量的, 但是比他自己的单核性能乘上64还少低好多, 比理论上限低更多了, 不知道是不是线程乱跑的原因, 多核我也没测试, 理论上不同核心是没有影响的. 最后社区的结果给的满准确.

一个简单的问题两个论文都错了, 所以论文不一定准确, 太坑了…还有就是Wiki其实目前没看到任何错误, 比一般论文还要靠谱?

还有就是我试了一下fmadd d15, d15, d16, d17这种标量的DP的运算其实理论上和双精度一样的(实测128核速度是900+GFLOPS附近), 只是需要的指令数量会少一点, 少占用一些其他资源. 所以说128bit的NEON是真的好弱…
